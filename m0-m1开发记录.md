## 3. 思考摘要与决策依据

### 3.1 为什么先做 M0，再做 M1

MVP 后续会涉及认证、权限、分类、客服账号、客户账号、工单、留言和状态推进。如果没有稳定的应用入口、配置、测试和静态页面骨架，直接开发业务功能会带来三个问题：

| 问题 | 风险 |
| --- | --- |
| 配置散落 | 数据路径、管理员初始化、Cookie 设置容易硬编码 |
| 测试缺口 | 后续数据和权限逻辑难以及时验证 |
| 目录不稳 | 业务代码容易互相耦合，后续替换 JSON 仓储会困难 |

因此 M0 先处理“项目能不能稳定运行”，M1 再处理“业务基础能不能可信保存和鉴权”。

### 3.2 为什么 M1 先做数据和仓储，再做会话和管理员

会话和管理员初始化都需要依赖用户数据保存能力。若先写会话逻辑而没有仓储边界，后续容易把业务代码直接绑定到 JSON 文件结构上，违背已确认的“仓储可替换”原则。

因此 M1 的实际顺序是：

```text
领域枚举/模型 -> JSON 存储 -> 仓储协议/实现 -> 密码哈希 -> 会话服务 -> 管理员初始化 -> 应用启动挂载
```

这个顺序对应实施计划中的任务依赖关系。

### 3.3 为什么服务端只保存会话摘要

已确认的安全底线要求密码不能明文保存，角色权限由后端校验。会话令牌虽然不是密码，但一旦原文落盘，被读取后也可直接冒用。

因此 M1 采用：

| 内容 | 处理方式 |
| --- | --- |
| 客户端会话令牌 | 只在创建会话时返回一次 |
| 服务端持久化 | 保存 SHA-256 摘要 |
| 验证过程 | 对请求令牌重新计算摘要后查找 |
| 撤销/过期 | 在服务端记录中处理 |

这样可以为后续 Cookie 登录和后端权限校验提供较稳的基础。

### 3.4 为什么 JSON 写入要做原子提交

MVP 使用 JSON 文件持久化，但业务写入常常不是单字段修改。例如后续创建工单时可能同时写入工单、首条留言和审计记录。如果写到一半失败，JSON 文件可能损坏或出现部分成功。

因此 M1 实现了：

| 机制 | 作用 |
| --- | --- |
| 进程内锁 | 控制本地单进程内并发写入 |
| 临时文件 | 先写完整新内容 |
| 原子替换 | 写入成功后再替换主文件 |
| 事务式 mutator | 让组合写入集中提交 |

这符合 MVP 的本地单进程演示边界，也为后续替换数据库适配器保留空间。

### 3.5 M1 是否使用了 TDD

M1 使用了更接近 TDD 的方式：

| 步骤 | 说明 |
| --- | --- |
| 先写测试 | 先为领域枚举、JSON 存储、密码、会话和管理员初始化编写测试 |
| 观察失败 | 测试因模块或行为缺失而失败，说明测试确实约束了待实现能力 |
| 再实现代码 | 按测试和实施计划补齐实现 |
| 回归验证 | 最终 `14 passed` |

需要说明的是，M1 不是形式化地为每一行代码都执行小步红绿重构，但核心行为采用了“测试先行、实现跟进”的方式。

### 3.6 为什么建议 M2 开新会话

M0-M1 主要是底座工作，信息密度尚可控制。M2 会开始进入真实管理接口、权限依赖、页面联动和更多测试，如果继续把所有讨论堆在同一会话中，容易出现：

| 风险 | 表现 |
| --- | --- |
| 上下文膨胀 | 新问题被旧讨论干扰 |
| 状态重复 | 已完成事项在对话中反复确认 |
| 记忆不稳定 | 某些阶段细节只存在聊天里，没有落地到文件 |
| 开发边界变模糊 | 容易不小心跨入 M3/M4 范围 |

更好的方式是将阶段事实沉淀为文件，然后在新会话中从项目文件恢复上下文。

## 6. 补充问答：多智能体编程

### Q6：你有多智能体编程或子智能体编程的用法吗？

回答摘要：

当前环境支持使用子智能体工具，适合做并行探索、并行编码和独立验证。但子智能体不应默认启用，通常应在用户明确要求“使用子智能体”或“多智能体推进”时使用。

推荐用法：

| 模式 | 用法 | 适合场景 |
| --- | --- | --- |
| 主智能体 | 负责总体设计、任务拆分、最终集成和验收 | 始终保留 |
| Explorer 子智能体 | 只读分析代码或文档，回答具体问题 | 查接口设计、权限规则、测试覆盖缺口 |
| Worker 子智能体 | 负责一个边界清晰的代码改动 | 不同模块并行开发 |
| Verifier 子智能体 | 独立跑测试、做代码审查、找遗漏 | 阶段收口前 |

核心原则：

- 子智能体只做边界清楚、互不重叠的任务。
- 主智能体负责最终集成、测试、验收和范围控制。
- 不把关键路径上的第一步交给子智能体后被动等待。
- 不让多个 Worker 同时修改同一批文件。
- 不用子智能体替代最终审查。

结合本项目进入 `M2` 的建议拆分：

| 智能体 | 任务 |
| --- | --- |
| 主智能体 | 设计 M2 实施顺序，负责认证上下文和最终集成 |
| Explorer 1 | 对照 API 设计和测试验收方案，整理 M2 管理接口验收点 |
| Worker 1 | 实现管理员认证依赖和权限错误处理 |
| Worker 2 | 实现分类管理服务/API |
| Worker 3 | 实现客服账号/客户启停相关服务/API |
| Verifier | 独立跑测试并检查是否越出 MVP 范围 |

阶段性建议：

`M2` 可以开始小规模尝试多智能体，先使用 `1` 个 Explorer 和 `1` 个 Worker。这样可以验证协作收益，同时避免过多并行导致冲突和范围失控。
