# M5-02 开发过程

## 1. 阶段锚定

依据 `docs/04-实施计划/01-MVP实施计划.md`，`M5` 为安全、交互与完整联调阶段。

`M5-02`：核查所有请求响应字段与权限拒绝行为。

`M5-02` 对应验收项：

| 验收项 | 说明 |
| --- | --- |
| `TC-SEC-001` ~ `TC-SEC-005` | 会话认证、角色越权、身份伪造、字段隔离安全用例 |
| `TC-SEC-010` | 错误响应符合接口约定 |
| `TC-SEC-011` | 页面和 API 不暴露非 MVP 安全增强能力 |

对应 API 安全要求：`API-SEC-001` ~ `API-SEC-008`、`API-C04` ~ `API-C07`。

## 2. 审查方法与依据

审查依据为已确认的 `docs/02-系统设计/03-API接口设计.md` 定义的响应字段边界和 `docs/03-测试与验收/01-测试与验收方案.md` 定义的安全用例。

审查方式：逐文件审查后端路由依赖注入、服务层身份来源、仓储层资源隔离和 Pydantic 响应模型输出字段，确认不与 API 设计约定的字段边界冲突。

## 3. 审查结果

### 3.5 TC-SEC-005：内部响应不暴露客户邮箱/密码哈希/会话字段

| 审查项 | 结果 |
| --- | --- |
| `user_brief_response`：只返回 `id/username`，无邮箱 | 通过 |
| `internal_ticket_list_item_response`：customer 用 `user_brief_response` | 通过 |
| `internal_ticket_detail_response`：customer 用 `user_brief_response` | 通过 |
| `agent_response`：含 `id/username/email/created_at`（API 设计文档约定） | 通过 |
| `customer_response`：含 `id/username/email/status/created_at`（API 设计文档约定） | 通过 |
| `current_user_response`：含 `id/username/email/role/status` | 通过 |
| `registered_user_response`：额外含 `created_at` | 通过 |
| 所有响应无 `password_hash` 字段 | 通过 |
| 所有响应无 `token_hash`、`expires_at`、`revoked_at` 字段 | 通过 |
| Cookie 原始值只在登录响应 `Set-Cookie` 头中一次性传输 | 通过 |

**审查范围**：`schemas/tickets.py`、`schemas/users.py`、`schemas/auth.py`、`schemas/categories.py`、全部 5 个服务文件。

**API 设计一致性说明**：
- API-SEC-005 要求"内部工单响应不得返回客户邮箱"：内部详情使用 `user_brief_response`（无 email），符合约定。
- 客服/客户管理列表响应包含 email，API 设计文档第 9.1/9.3 节明确约定了 email 字段，与管理场景需求一致。
- API-SEC-006 要求"用户管理响应不得返回密码哈希或会话信息"：`agent_response` 和 `customer_response` 均不包含这些字段。

**结论：通过**

## 4. 权限与字段覆盖总览

| API 模块 | 认证依赖 | 角色依赖 | 响应字段隔离 | 请求体伪造防护 |
| --- | --- | --- | --- | --- |
| `/auth` | `get_current_user`（logout/me） | 无（logout/me 任意角色） | 无敏感字段 | `extra="forbid"` |
| `/categories/active` | `require_customer` | 仅客户 | 仅 id/name | `extra="forbid"` |
| `/admin/categories` | `require_admin` | 仅管理员 | 含 created_by_user_id | `extra="forbid"` |
| `/admin/agents` | `require_admin` | 仅管理员 | 无 password_hash | `extra="forbid"` |
| `/admin/customers` | `require_admin` | 仅管理员 | 含 email（管理需要） | `extra="forbid"` |
| `/customer/tickets` | `require_customer` | 仅客户 + 资源归属 | 无 assignee/audit_logs | `extra="forbid"` |
| `/internal/tickets` | `require_internal_user` | agent/admin | customer 无 email | `extra="forbid"` |

## 5. 范围控制

本阶段只处理响应字段隔离、权限拒绝行为和错误响应契约的静态审查。

未扩大范围：
- 未做用户文本 XSS 安全渲染复核，该工作留给 `M5-03`。
- 未执行 UI 完整联调，该工作留给 `M5-04`。
- 未执行浏览器矩阵验证，该工作留给 `M5-05`。

## 6. M5-02 阶段结论

`M5-02` 核查所有请求响应字段与权限拒绝行为已完成。

已满足：
- API 字段隔离符合已确认的 API 接口设计文档约定。
- 客户工单响应不包含负责人标识或内部操作记录。
- 内部工单响应不包含客户邮箱。
- 所有用户管理响应不包含密码哈希或会话信息。
- 所有请求模型拒绝额外字段注入。
- 所有写操作身份来源为服务端会话，不接受请求体伪造。
- 角色权限拒绝返回 403，前端按钮隐藏不替代后端校验。
- 错误响应契约已由 `test_error_response_contract.py` 覆盖。
- 页面和 API 不暴露密码找回、多因素认证、验证码、登录锁定等非 MVP 能力。
- 所有审查项通过，未发现字段泄露或权限校验缺失。

下一步可进入 `M5-03`：确保所有用户文本均以文本形式展示（XSS 防护）。
