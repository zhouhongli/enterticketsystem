## 1. M2 相关问答记录

### Q1：请你来设计下，如何运用多智能体进行开发。

回答摘要：

建议采用“主智能体控场 + 少量子智能体并行推进 + 严格集成门禁”的方式，而不是简单增加智能体数量。核心目标是让任务并行但不失控。

#### 1.1 多智能体角色设计

| 角色 | 职责 | 是否改代码 |
| --- | --- | --- |
| 主智能体 | 读基线、拆任务、做关键路径、集成、最终验收 | 是 |
| Explorer | 只读分析文档/代码，提炼验收点、接口规则、风险 | 否 |
| Worker | 负责一个明确模块或文件范围的实现 | 是 |
| Verifier | 独立跑测试、做审查、检查是否越界 | 通常否 |

结论：

主智能体必须保留最终判断权。子智能体可以提速，但不能替代范围控制、集成和验收。

#### 1.2 M2 推荐使用方式

`M2` 建议先小规模使用多智能体，不要一开始就开启过多并行任务。

| 阶段 | 主智能体任务 | 子智能体任务 |
| --- | --- | --- |
| M2 准备 | 读取 `AGENTS.md`、实施计划、API 设计、测试方案，确认 M2 边界 | Explorer 整理 M2 接口、权限、测试验收清单 |
| M2-01 | 主智能体本地实现管理员认证上下文，因为它是后续管理接口的关键路径 | 不并行写这块 |
| M2-02 / M2-03 | 主智能体集成认证依赖和路由注册 | Worker A 实现分类服务、API 和测试 |
| M2-04 / M2-05 | 主智能体处理共享用户仓储与冲突点 | Worker B 实现客服账号、客户启停服务、API 和测试 |
| M2-06 | 主智能体做前端页面联调和浏览器验证 | Verifier 独立跑测试、检查 MVP 范围和权限漏洞 |

#### 1.3 文件所有权设计

为了减少冲突，进入 M2 前应先划清写入范围。

| 负责人 | 写入范围 |
| --- | --- |
| 主智能体 | `backend/app/api/dependencies.py`、通用错误处理、路由注册、最终集成 |
| Worker A | `backend/app/schemas/categories.py`、`backend/app/services/category_service.py`、`backend/app/api/routes/categories.py`、分类相关测试 |
| Worker B | `backend/app/schemas/users.py`、`backend/app/services/user_admin_service.py`、`backend/app/api/routes/admin_users.py`、用户管理相关测试 |
| 主智能体 | `frontend/pages/internal/categories.html`、`frontend/pages/internal/agents.html`、`frontend/pages/internal/customers.html`、共享 JS 调整 |

原则：

- 多个 Worker 不同时修改同一批文件。
- 共享仓储、认证依赖、错误模型等关键公共边界由主智能体统一处理。
- Worker 应被明确告知：不能回退其他人的修改，必须适配已有代码。

#### 1.4 执行节奏

推荐流程：

1. 主智能体先写本阶段任务板：目标、边界、文件所有权、测试门槛。
2. Explorer 并行整理验收清单，不改代码。
3. 主智能体完成关键路径，即认证上下文和管理员权限依赖。
4. Worker 只接收边界清楚的任务，并被明确告知不要改其他人的文件。
5. Worker 返回后，主智能体审查变更、合并思路、补齐缺口。
6. Verifier 独立检查测试、接口权限、MVP 范围和潜在遗漏。
7. 主智能体跑全量测试和本地浏览器验证，再更新阶段记录。

#### 1.6 M2 多智能体门禁规则

| 门禁 | 要求 |
| --- | --- |
| 范围门禁 | 只做当前里程碑，不提前做 M3/M4 |
| 权限门禁 | 管理接口必须后端校验管理员角色 |
| 测试门禁 | Worker 自己跑相关测试，主智能体最终跑全量测试 |
| 集成门禁 | 主智能体统一处理冲突、路由注册、错误模型一致性 |
| 文档门禁 | 阶段完成后更新 `AGENTS.md` 和阶段开发记录 |

阶段性结论：

`M2` 建议采用 `1` 个 Explorer + `1` 个 Worker 起步。等协作节奏跑顺后，再在 `M3` 或 `M4` 这种业务切片更清晰的阶段增加 Worker 数量。

### Q3：M2 采用 1 个 Explorer + 1 个 Worker。

确认结论：

用户确认 `M2` 阶段采用小规模多智能体协作方式：`1` 个 Explorer + `1` 个 Worker。

执行口径：

| 角色 | M2 初始职责 |
| --- | --- |
| 主智能体 | 负责读取基线、确认 M2 边界、完成关键路径、集成子智能体结果、运行最终测试与验收 |
| Explorer | 只读整理 M2 接口、权限、测试验收清单和潜在风险，不修改代码 |
| Worker | 负责一个边界清晰的实现切片，优先建议从分类管理服务/API/测试开始 |

协作约束：

- 主智能体先完成或至少稳定 `M2-01` 管理员认证上下文和管理员路由授权依赖。
- Explorer 与 Worker 的任务必须互不重叠，避免重复分析或修改同一批文件。
- Worker 的写入范围需要在派发前明确，不得修改认证依赖、共享仓储公共边界、前端页面或其他未授权文件。
- Worker 返回后，由主智能体统一审查、集成、补测和判断是否符合 MVP 范围。
- 后续若要增加第二个 Worker，应在完成一次小规模协作复盘后再决定。

### Q6：M2 多智能体运行的整体架构和流程是什么？“用户管理切片”是什么意思？一个切片是否对应一个智能体？如何理解 Edge headless？

回答摘要：

`M2` 的多智能体运行不是多个智能体随意并行写代码，而是“主智能体控场，子智能体做边界清楚的辅助任务”。主智能体负责读基线、确定阶段边界、控制关键路径、集成结果、跑最终测试和更新记录；Explorer 只读分析；Worker 负责一个明确实现范围。

整体架构：

| 层级 | 角色 | M2 中的实际工作 |
| --- | --- | --- |
| 控制层 | 主智能体 | 负责 M2 目标拆解、管理员认证关键路径、用户管理切片、集成、测试、文档更新 |
| 分析层 | Explorer | 只读整理 M2 API、权限、测试清单、风险，不改文件 |
| 执行层 | Worker | 实现分类管理切片，只改被授权的四类文件 |
| 验收层 | 主智能体 | 审查子智能体结果，合并到主线，跑全量测试和页面检查 |

执行流程：

1. 用户确认 `M2` 采用 `1` 个 Explorer + `1` 个 Worker。
2. 主智能体启动 Explorer，让它只读分析 M2 接口、权限、测试和风险。
3. 主智能体不等待 Explorer，先推进关键路径 `M2-01` 管理员认证上下文。
4. Explorer 返回风险提醒：响应不得泄露 `password_hash`/会话字段；禁用客户应主动撤销旧会话。
5. 主智能体稳定公共边界：认证依赖、错误响应、仓储扩展、审计写入、响应白名单。
6. 主智能体派发 Worker 处理“分类管理切片”，并明确写入范围，避免冲突。
7. 主智能体并行处理不重叠的“用户管理切片”。
8. Worker 返回后，主智能体统一审查、集成、修正兼容性、跑全量测试。
9. 主智能体重启本地服务，用 Edge headless 做页面加载检查，并更新阶段记录。

“切片”的含义：

切片是一个可独立开发、测试和验收的功能工作包。它不一定等于一个完整模块，也不一定等于一个智能体。切片的重点是边界清楚、输入输出清楚、文件范围清楚、测试目标清楚。

M2 中的切片示例：

| 切片 | 包含内容 | 负责人 |
| --- | --- | --- |
| 管理员认证切片 | 当前用户识别、管理员权限依赖、未登录/无权限错误 | 主智能体 |
| 分类管理切片 | 分类列表、新增、编辑、启停、有效分类查询、分类测试 | Worker |
| 用户管理切片 | 客服账号列表/创建、客户列表/启停、禁用客户撤销会话、用户管理测试 | 主智能体 |
| 管理页面切片 | 分类/客服/客户三张页面调用 API 和展示反馈 | 主智能体 |

“用户管理切片”不是完整的用户系统。它只指 `M2` 范围内的管理员用户管理能力，即：

- 管理员查看客服账号列表。
- 管理员创建客服账号。
- 管理员查看客户账号列表。
- 管理员启用/禁用客户账号。
- 禁用客户时撤销该客户现有会话。
- 接口响应不得返回密码哈希或会话字段。

它不包含客户注册、登录、退出、找回密码、修改密码、客服禁用、客服编辑、管理员创建等能力，这些要么在其他阶段，要么不属于 MVP。

一个切片是否对应一个智能体：

不一定。更准确地说，一个切片可以分配给一个智能体，但不是强制一一对应。

| 情况 | 是否合适 |
| --- | --- |
| 一个 Worker 负责一个边界清楚的切片 | 推荐 |
| 主智能体自己处理关键路径切片 | 推荐 |
| 一个 Worker 同时处理多个不相关切片 | 不推荐，容易扩大范围 |
| 多个 Worker 同时改同一个切片 | 不推荐，容易冲突 |
| Explorer 负责一个实现切片 | 不适合，Explorer 只读分析 |

在 M2 中，Worker 只负责“分类管理切片”；主智能体负责“管理员认证切片”和“用户管理切片”。这说明切片是任务组织方式，不是智能体数量单位。

Edge headless 的含义：

Edge headless 是“无界面的 Microsoft Edge 浏览器”。它不会弹出浏览器窗口，但会像真实浏览器一样加载页面、执行 JavaScript、请求后端接口，并可以输出页面 DOM。

在 M2 中使用 Edge headless 的目的：

| 检查点 | 作用 |
| --- | --- |
| 页面能否加载 | 确认 HTML、CSS、JS 路径没有断 |
| JS 是否执行 | 确认前端脚本没有语法错误或模块加载错误 |
| 未登录反馈 | 确认调用受保护 API 后页面能显示“请先登录管理员账号” |
| 静态入口状态 | 确认三张管理页可被本地服务访问 |

Edge headless 不是完整人工验收。它更像一次轻量浏览器烟雾测试，用来快速发现页面打不开、脚本崩溃、API 调用失败但页面没有反馈这类问题。M2 因为还没有 M3 登录接口，所以它只能验证“页面加载与未登录反馈”，不能完成“管理员真实登录后操作分类/客服/客户”的完整浏览器流程。

### Q7：为什么称为 M0、M1、M2，这里的 M 是指什么？

回答摘要：

这里的 `M` 指 `Milestone`，即“里程碑”。`M0`、`M1`、`M2` 不是月份，也不是模块编号，而是按交付顺序划分的阶段性目标。

命名含义：

| 名称 | 含义 | 在本项目中的定位 |
| --- | --- | --- |
| `M0` | Milestone 0，准备性里程碑 | 工程初始化与运行骨架 |
| `M1` | Milestone 1，第一个基础能力里程碑 | 数据、仓储、安全与会话基础 |
| `M2` | Milestone 2，第二个业务能力里程碑 | 管理基础配置能力 |
| `M3` | Milestone 3 | 客户认证与工单能力 |
| `M4` | Milestone 4 | 内部处理闭环 |
| `M5` | Milestone 5 | 安全、交互与完整联调 |
| `M6` | Milestone 6 | 验收准备与交付 |

为什么从 `M0` 开始：

`M0` 通常表示正式业务功能开发前的工程准备阶段。它不一定交付完整业务功能，但它提供后续所有里程碑依赖的基础，例如项目结构、运行入口、配置、测试框架和静态页面骨架。

在本项目中，`M0 -> M1 -> M2` 的逻辑是：

```text
M0：先让项目能跑、能测、能扩展
M1：再让数据、安全、会话这些底座可信
M2：然后开始做第一个管理侧业务能力
```

这样命名的好处：

- 每个阶段都有明确交付物。
- 每个阶段都有完成门槛。
- 后续沟通可以直接说“进入 M3”或“M2 未完成”，减少歧义。
- 任务依赖关系更清楚，例如 `M2` 的分类能力会阻塞 `M3` 的客户建单能力。

### Q8：下面你觉得，我们可以做什么？

回答摘要：

建议下一步进入 `M3` 客户认证与工单能力，并优先从 `M3-01` 认证接口与登录/注册页面开始。

原因：

`M2` 已经完成管理接口和管理页面接线，但还缺真实登录、注册、退出和当前身份接口。没有 `M3-01`，管理员页面只能显示未登录提示，不能通过真实浏览器登录后完成分类、客服账号和客户账号管理操作。

推荐执行顺序：

| 顺序 | 任务 | 说明 |
| --- | --- | --- |
| 1 | `M3-01` 认证接口与登录/注册页面 | 实现注册、登录、退出、当前身份接口；登录后按角色跳转 |
| 2 | `M3-02` 客户建单接口 | 客户选择有效分类，创建待分配工单，保存分类快照和创建记录 |
| 3 | `M3-03` / `M3-04` 客户工单列表、详情、留言 | 客户只能查看自己的工单；未关闭工单可留言；不暴露负责人和内部记录 |
| 4 | `M3-05` 客户门户页面联调 | 注册、登录、建单、查看、留言形成客户侧闭环 |

建议：

从工程节奏看，`M3` 是一个适合开启新会话的边界。新会话可以使用以下提示开始：

```text
请阅读 AGENTS.md、README.md、M2开发过程.md 和 docs/04-实施计划/01-MVP实施计划.md，从 M3-01 认证接口与登录/注册页面开始继续开发，不扩大 MVP 范围。
```

结论：

优先进入 `M3-01`。完成认证接口后，`M2` 的管理页面可以通过真实管理员登录进行完整联调，也能为客户建单和客户门户能力提供统一身份基础。
