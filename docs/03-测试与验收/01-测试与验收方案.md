## 1. 文档目的

本文档将已确认的产品需求、软件需求、页面交互要求及系统设计约束转化为可执行、可判定的测试与验收规则。其目标是验证企业售后工单系统 MVP 能够支持个人客户售后问题从提交到关闭的完整闭环，并在角色授权、敏感数据保护、服务端会话、JSON 持久化与未来存储替换边界方面符合基线。

本方案在代码实现前确定验收口径。方案确认后，开发过程应依据其中的测试场景组织自动化测试和人工验收；若实施中需要改变本方案对应的已确认需求或设计，必须先完成变更确认，不得通过降低验收标准绕过基线。

## 3. 测试目标与验收原则

### 3.1 测试目标

| 目标 | 验证重点 |
| --- | --- |
| 业务闭环有效 | 客户建单、管理员分配、客服处理、关闭和历史查看可连续演示完成。 |
| 权限边界可靠 | 客户资源归属、客服负责人限制、管理员专属操作均由后端强制执行。 |
| 状态与数据一致 | 工单只按线性状态推进，关键变更保留记录，不发生部分保存。 |
| 凭证与信息安全 | 密码、会话凭证和非授权业务字段不被存储或响应不当暴露。 |
| 本地持久化可用 | JSON 数据在服务重启后可继续读取，单进程写入保护有效。 |
| 使用反馈清晰 | 页面呈现身份、状态、操作结果和错误信息，核心流程可顺畅使用。 |

### 3.2 验收原则

| 编号 | 原则 |
| --- | --- |
| TP-01 | 所有标记为“必须”的 SRS 需求均应具有测试覆盖，不能以未测试方式通过验收。 |
| TP-02 | API 测试结果是权限和数据暴露边界的最终判定依据，前端隐藏按钮不能替代后端验证。 |
| TP-03 | 测试数据应可重置、可重复执行，不以人工修改数据文件制造通过结果。 |
| TP-04 | 严重影响业务闭环、权限、安全或数据一致性的问题未关闭时，MVP 不得验收通过。 |
| TP-05 | MVP 非目标不作为验收缺陷，也不得在实现中被默认为必交付能力。 |

## 4. 测试范围

### 4.1 纳入测试的能力

| 范围 | 包含内容 |
| --- | --- |
| 认证与账号 | 客户注册，三类用户登录/退出，首个管理员初始化，客服创建，客户启用/禁用。 |
| 分类管理 | 分类新增、编辑、启用/停用，有效分类选择及分类历史显示。 |
| 客户工单 | 创建工单、我的工单列表、详情查看、公开留言。 |
| 内部处理 | 全部工单列表、单一状态筛选、详情查看、分配/重新分配、公开留言、状态推进。 |
| 状态闭环 | `待分配 -> 处理中 -> 已解决 -> 已关闭`，以及所有禁止路径。 |
| 操作与历史 | 工单创建、分配/重新分配、状态变化、分类变化、客户启停的记录与历史快照。 |
| 安全与授权 | 会话 Cookie、密码哈希、字段隔离、脚本内容展示、角色和资源授权。 |
| 数据持久化 | 单一版本化 JSON 文件、原子保存、重启读取、仓储边界检查。 |
| 页面体验 | 角色导航、必填与错误反馈、空状态、确认提示、主流桌面浏览器基本流程。 |

### 4.2 不纳入 MVP 验收的能力

| 非范围能力 | 验收处理方式 |
| --- | --- |
| 附件上传、外部通知、SLA、统计报表 | 不要求实现；若页面或接口将其作为正式能力提供，应进行范围审查。 |
| 高级搜索、多条件筛选、批量操作、导出 | 不要求实现；内部列表仅验证单一状态筛选。 |
| 客户确认关闭、取消、退回、重开或自动关闭 | 不要求实现；出现对应业务操作入口视为与基线不一致。 |
| 密码找回、多因素认证、验证码、登录锁定 | 不作为 MVP 通过条件。 |
| 真实关系型数据库、迁移程序、生产部署和高并发 | 不执行功能验收；只审查仓储可替换边界未被破坏。 |
| 精确性能指标和压力容量 | 不设硬性阈值；仅检查目标小规模数据下无明显阻塞使用的问题。 |

## 5. 测试策略

### 5.1 测试层次与方式

| 测试层次 | 方法 | 主要内容 | 预期执行阶段 |
| --- | --- | --- | --- |
| 规则单元测试 | 自动化 | 输入校验、状态转换、授权判断、历史快照、密码与会话辅助逻辑 | 开发中持续执行 |
| 仓储与服务测试 | 自动化 | JSON 读写、原子更新、审计同事务保存、重启载入、仓储协议替换性 | 后端实现完成后 |
| API 集成测试 | 自动化为主 | 各路由正常响应、错误码、Cookie、字段隔离、角色越权与资源归属 | 接口实现完成后 |
| 页面联调测试 | 浏览器人工/可自动化辅助 | 导航、表单反馈、按钮可见性、详情展示、脚本安全呈现 | 前后端联调后 |
| 业务验收测试 | 人工演示并留存结果 | 从配置、注册、建单、处理到关闭的主流程和关键异常流程 | MVP 提交验收时 |
| 文档/代码审查 | 静态核对 | 仓储抽象、单进程约束、非范围功能、配置与敏感日志规则 | 实施完成前 |

### 5.2 自动化与人工测试分工

| 范围 | 自动化要求 | 人工验收要求 |
| --- | --- | --- |
| API 正常与错误路径 | 关键接口和权限冲突必须自动化覆盖。 | 抽样核对页面触发结果。 |
| 状态、分配、留言规则 | 必须自动化覆盖允许与禁止动作。 | 演示完整关闭闭环。 |
| 会话与字段隔离 | 必须通过接口响应和存储检查验证。 | 浏览器核对身份与退出体验。 |
| UI 展示和确认交互 | 可使用浏览器测试辅助。 | 必须检查核心页面、反馈、确认框和中文状态显示。 |
| 原子保存与重启读取 | 后端自动化或受控故障测试。 | 演示重启后读取既有数据。 |

## 6. 测试环境与数据准备

### 6.1 环境要求

| 项目 | 验收环境要求 |
| --- | --- |
| 应用结构 | 单个 FastAPI 应用同时提供静态前端与 `/api/v1` 接口，同源访问。 |
| Python 环境 | 使用实施计划最终确定并锁定的 Python 版本与依赖清单。 |
| 数据存储 | 独立测试数据目录中的版本化 `store.json`，测试前可恢复为已知初始状态。 |
| 运行边界 | 仅启动一个应用进程执行写入测试，不将多进程写入作为 MVP 能力。 |
| 时间基准 | 持久化数据按 UTC ISO 8601 检查；页面时间以实际实现的本地展示为准。 |
| 桌面浏览器 | 验收时使用当日最新稳定版 Google Chrome 与 Microsoft Edge 各执行一次核心流程。 |
| Cookie 环境 | 本地 HTTP 可配置 `Secure=false`；须验证 `HttpOnly`、`SameSite=Lax`、`Path=/` 和 8 小时期限。 |

### 6.2 基础测试数据

| 标识 | 测试数据 | 用途 |
| --- | --- | --- |
| `ADMIN-01` | 初始化管理员 `admin01` | 管理、分配和管理员直接处理测试 |
| `AGENT-01` | 启用客服 `agent01` | 被分配后的处理测试 |
| `AGENT-02` | 启用客服 `agent02` | 非负责人和重新分配测试 |
| `CUS-01` | 启用客户 `customer01` | 建单与本人操作测试 |
| `CUS-02` | 启用客户 `customer02` | 他人工单越权测试 |
| `CUS-03` | 后续被禁用客户 `customer03` | 禁用与会话失效测试 |
| `CAT-01` | 启用分类“产品故障” | 正常建单 |
| `CAT-02` | 停用分类“历史咨询” | 停用与历史保留测试 |
| `TKT-01` | `CUS-01` 新建、待分配工单 | 分配与管理员直接处理测试 |
| `TKT-02` | 分配给 `AGENT-01` 的处理中工单 | 留言、重新分配、状态测试 |
| `TKT-03` | 已关闭工单 | 关闭后禁止操作测试 |

测试账号密码仅用于测试环境，实际存储检查必须验证仅保存安全哈希，文档和测试报告不得记录有效会话 Cookie 原始值。

## 8. 功能与业务流程测试用例

### 8.1 认证与账号

| 用例编号 | 优先级 | 测试场景与操作摘要 | 预期结果 | 覆盖需求 |
| --- | --- | --- | --- | --- |
| `TC-AUTH-001` | 关键 | 访客以合法用户名、邮箱、密码注册客户。 | 创建启用的客户账号；不自动登录；提示登录。 | `SRS-AUTH-001` ~ `003`、`UI-C02` |
| `TC-AUTH-002` | 重要 | 以重复用户名、大小写不同但相同邮箱或首尾空白标识注册。 | 注册失败并提示标识已使用；不新增账号。 | `SRS-AUTH-004` |
| `TC-AUTH-003` | 重要 | 密码与确认密码不一致，或密码长度无效。 | 展示字段错误；不新增账号。 | `SRS-AUTH-005` |
| `TC-AUTH-004` | 关键 | 客户、客服、管理员分别以用户名或邮箱登录。 | 成功创建会话；按角色进入对应首页。 | `SRS-AUTH-006`、`007`、`NFR-003` |
| `TC-AUTH-005` | 关键 | 不存在账号、错误密码、禁用客户分别登录。 | 均返回统一失败提示且不建立会话。 | `SRS-AUTH-008`、`009`、`019` |
| `TC-AUTH-006` | 关键 | 已登录用户退出后再次访问受保护接口和页面。 | 当前会话撤销/清除，后续访问要求登录。 | `SRS-AUTH-010` |
| `TC-AUTH-007` | 重要 | 检查访客界面和接口能力。 | 不提供找回密码、自助修改密码、邮箱验证或登录锁定功能。 | `SRS-AUTH-011` |
| `TC-AUTH-008` | 关键 | 空存储启动并按配置初始化管理员；已有管理员后再次启动。 | 首次创建管理员，后续不重复创建或覆盖；访客不能建管理员。 | `SRS-AUTH-012`、`TD-C08` |
| `TC-AUTH-009` | 关键 | 管理员创建客服；客户或客服尝试创建客服。 | 管理员成功创建启用客服；其他角色被拒绝。 | `SRS-AUTH-013` ~ `015` |
| `TC-AUTH-010` | 重要 | 查看客服管理界面及接口。 | MVP 无编辑、禁用或重置客服密码能力。 | `SRS-AUTH-016` |
| `TC-AUTH-011` | 关键 | 管理员查询客户并禁用/重新启用客户。 | 状态正确变化并记录；仅管理员可执行。 | `SRS-AUTH-017`、`018` |
| `TC-AUTH-012` | 关键 | 禁用已有工单/留言的客户并使用其已有会话访问。 | 会话下一次请求失效；客户不能继续操作；历史数据仍供管理员查询。 | `SRS-AUTH-019`、`020` |

| `TC-TKT-005` | 关键 | `CUS-02` 直接请求 `CUS-01` 的工单资源。 | 返回不暴露资源存在性的拒绝结果，且不返回内容。 | `SRS-TKT-008`、`SEC-002`、`004` |
| `TC-TKT-006` | 关键 | 客服和管理员查看内部列表、详情及按单一状态筛选。 | 可查看全部工单及规定字段、留言和基础操作记录；筛选正确。 | `SRS-TKT-009` ~ `011` |

### 8.3 分配、留言与状态闭环

| 用例编号 | 优先级 | 测试场景与操作摘要 | 预期结果 | 覆盖需求 |
| --- | --- | --- | --- | --- |
| `TC-ASG-001` | 关键 | 管理员把未关闭工单分配给有效客服。 | 负责人更新；产生分配记录；工单状态不自动变化。 | `SRS-ASG-001`、`002`、`004`、`005`、`007` |
| `TC-ASG-002` | 关键 | 管理员将处理中工单从 `AGENT-01` 改派给 `AGENT-02`。 | 负责人更新并记录前后值；原客服立即失去处理权限。 | `SRS-ASG-002`、`005`、`006` |
| `TC-ASG-003` | 关键 | 非管理员分配工单，或管理员将工单分给客户/管理员/不存在账号。 | 请求拒绝，原负责人不变且无成功记录。 | `SRS-ASG-001`、`004` |
| `TC-ASG-004` | 关键 | 管理员尝试重新分配已关闭工单。 | 请求冲突；负责人不变。 | `SRS-ASG-003`、`STS-008` |
| `TC-MSG-001` | 关键 | 客户在本人的未关闭工单留言。 | 留言保存并可查看；发送人和时间正确；状态不变化。 | `SRS-MSG-001`、`002`、`008` |

## 9. 权限、安全与接口验收用例

| 用例编号 | 优先级 | 测试场景与操作摘要 | 预期结果 | 覆盖需求/设计 |

## 14. 缺陷分级与验收处理

| 等级 | 定义 | 示例 | 验收处理 |
| --- | --- | --- | --- |
| 严重 | 核心流程不可完成、安全/权限失守或数据丢失损坏。 | 越权读取他人工单；明文密码；关闭后仍可写入；JSON 损坏。 | 必须修复并复测通过。 |
| 高 | 关键业务规则不符合基线，存在明显错误结果或无法继续主要操作。 | 客服不能处理被分配工单；状态可跳级；退出后仍可访问。 | 必须修复并复测通过。 |
| 中 | 非核心路径行为错误或重要交互反馈缺失，但存在可接受绕行方式。 | 状态筛选错误；确认提示遗漏。 | 原则上修复；未修复需用户明确接受。 |
| 低 | 不影响规则和流程的展示问题或优化建议。 | 对齐、文字微小问题。 | 可记录后验收，需列入已知问题。 |
