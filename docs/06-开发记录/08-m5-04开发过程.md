## 1. 阶段锚定

依据 `docs/04-实施计划/01-MVP实施计划.md`，`M5-04`：依据页面交互说明与低保真确认结论，补齐确认提示、空状态、排序、导航和无权限页面表现。

`M5-04` 对应验收项：

| 验收项 | 说明 |
| --- | --- |
| `TC-UI-001` ~ `TC-UI-008` | UI 完整性：角色导航、错误反馈、空状态、状态中文、确认提示、排序、状态筛选、权限拒绝 |

对应页面实现约束：

| 约束 | 来源 |

### 2.2 TC-UI-002：表单必填缺失、长度错误及后端业务错误反馈

**前端校验覆盖**：

| 页面 | 校验规则 | 实现 |
| --- | --- | --- |
| 登录页 | identifier/password 非空 | `auth-pages.js:65-68` |
| 注册页 | 用户名长度、邮箱格式、密码长度、密码匹配 | `auth-pages.js:93-110` |
| 新建工单 | 分类非空、标题 1-100、描述 1-4000 | `customer-tickets.js:133-148` |
| 留言（客户） | 内容 1-2000 | `customer-tickets.js:310-316` |
| 留言（内部） | 内容 1-2000 | `internal-tickets.js:393-398` |
| 分配工单 | 负责人非空 | `internal-tickets.js:409-413` |

**后端错误反馈链路**：

```
后端 422 → api-client.js buildApiError()
           → codeMessages.VALIDATION_ERROR + fieldErrorSummary()
           → 前端展示："填写的信息不符合要求。请检查：XX、XX。"

后端 401/403/404/409 → api-client.js buildApiError()
                      → codeMessages 映射到中文错误说明
                      → 前端展示对应中文错误

后端网络错误 → api-client.js catch → "网络连接失败，请检查本地服务是否正在运行。"
```

**非敏感输入保留**：
- HTML `<form>` 提交失败时不执行 `form.reset()`，表单值自然保留。
- 密码字段通过 `clearPasswords()` 专门清空（`auth-pages.js:87-91`）。

**结论：通过 ✅**

### 2.7 TC-UI-007：内部列表状态筛选与无高级筛选

**筛选选项**（`internal/tickets.html:23-28`）：
- 5 个固定选项：全部、待分配、处理中、已解决、已关闭 ✅

**默认行为**：
- 不带 `status` 参数 → 请求全部工单 ✅
- `currentStatusFilter()` 返回 `""` → 路径 `/api/v1/internal/tickets` ✅

**状态筛选切换**：
- 点击链接 → URL 携带 `?status=xxx` → `initTicketList()` 读取参数 → 请求 `/api/v1/internal/tickets?status=xxx` ✅
- 服务端只接受 `status` 参数，拒绝其他查询参数 ✅

**活跃状态高亮**：
- `renderFilters()` 对比 `data-status-filter` 与当前值 → 添加 `active` class ✅
- CSS `.filter.active` 定义高亮样式（accent 边框 + 背景 + 粗体）✅

**无高级筛选**：
- 无搜索框、无多条件组合、无分页、无导出、无批量操作入口 ✅

**不支持参数被拒绝**：
- `internal.py:23-29` `reject_unsupported_query_params(request, {"status"})` ✅
- 多值 `status` 也被拒绝（`internal.py:40-45`）✅

**结论：通过 ✅**

### 2.8 TC-UI-008：无权限访问页面表现

**会话未认证（401）**：

| 场景 | 行为 | 实现 |
| --- | --- | --- |
| 访客访问任何受保护页面 | `sessionReady` 调 `/auth/me` 返回 401 → 跳转登录页 | `session-ui.js:111-113` |
| 会话过期用户访问 | 同上 → 跳转登录页 | 同上 |

**角色不匹配**：

| 场景 | 行为 | 实现 |
| --- | --- | --- |
| 客户访问内部页面 | `isAllowedOnCurrentPage` false → 跳转客户首页 | `session-ui.js:103-105` |
| 客服访问客户页面 | `isAllowedOnCurrentPage` false → 跳转内部首页 | `session-ui.js:103-105` |
| 管理员访问客户页面 | 同上 → 跳转内部首页 | 同上 |

**跨角色页面脚本二次拦截**：

| 文件 | 拦截行为 |
| --- | --- |
| `customer-tickets.js:365` | `user.role !== "customer" → return` 中止业务逻辑 |
| `internal-tickets.js:524` | `!["agent", "admin"].includes(user.role) → return` 中止 |

**客户越权详情**：

| 场景 | 行为 |
| --- | --- |
| 客户请求他人工单 | API 返回 404（不暴露资源存在性）→ 前端显示"工单不存在或无权访问。" |

**后端强校验**（即使前端被绕过）：

| 场景 | 后端行为 |
| --- | --- |
| 客户请求内部路由 | `require_internal_user` → 403 |
| 客户请求他人工单 | 仓储层按 `customer_user_id` 过滤 → 查无结果 → 404 |
| 非管理员请求管理路由 | `require_admin` → 403 |
| 客服非负责人留言/改状态 | 仓储层校验 `assignee_user_id` → `"forbidden"` → 403 |

**结论：通过 ✅**

`M5-04` UI 完整联调审查已完成。
