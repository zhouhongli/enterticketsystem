## 1. 制作定位

本演示文稿不是企业售后工单系统的产品汇报，也不是 FastAPI、前端或 JSON 存储的技术教程。它是一套案例型培训材料，用一个真实项目的构建过程，讲清楚 AI Agent 协作下软件项目如何从模糊想法走向开发前基线。

课程要避免两种偏差：

1. 只讲项目功能，把培训变成“售后工单系统需求说明会”。
2. 只讲 AI Agent 工具能力，把培训变成“工具宣传或操作演示”。

真正要讲的是：当一个讲师、产品负责人或项目负责人只有一个初始想法时，如何与 Agent 配合，通过逐步澄清、确认、落盘、追踪和验证，把想法推进到可以开工开发的状态。

建议课程标题为：

```text
从一个想法到可开发方案
AI Agent 协作下的软件项目构建全过程
以企业售后工单系统 MVP 为例
```

## 2. 教学目标

学员完成本课程后，应能形成以下认识：

| 目标 | 学员应掌握的能力 |
| --- | --- |
| 理解前期构建过程 | 能说清一个软件想法在编码前需要经历哪些关键确认和产物沉淀。 |
| 建立 MVP 边界意识 | 能区分“首期必须做”和“未来可能做”，避免项目初期范围膨胀。 |
| 理解文档分工 | 能说明 PRD、SRS、页面交互、技术设计、数据模型、API 设计、测试方案、实施计划各自回答什么问题。 |
| 理解追踪关系 | 能用一个规则示例说明如何贯穿需求、页面、API、测试和实施计划。 |
| 理解 Agent 协作边界 | 能说明 Agent 可以帮助生成、整理和检查，但不能替代人的业务取舍和确认责任。 |
| 迁移到自身项目 | 能用同样节奏为自己的软件想法设计一组澄清问题和 MVP 边界。 |

## 5. 开场与课程边界

### `P01` 封面：从一个想法到可开发方案

**页面目的：** 建立课程主题，明确这是一个软件项目构建过程复盘，而不是产品功能介绍。

**页面可见文本建议：**

```text
从一个想法到可开发方案

AI Agent 协作下的软件项目构建全过程
以企业售后工单系统 MVP 为例

本课关注：
想法澄清 | 需求基线 | 设计基线 | 测试验收 | 实施计划
```

**画面结构：**

- 页面中心放课程主标题。
- 下方放五节点流程线：`想法 -> 澄清 -> 基线 -> 验证 -> 计划`。
- 右下角放目标听众：`产品经理 / 项目经理 / 研发管理人员 / 业务分析人员`。

**讲解重点：**

开场要先告诉学员，本课不是展示“已经做好的系统”，而是拆解“系统是怎么一步步被定义出来的”。软件开发最危险的地方往往不是编码慢，而是太早编码，把未确认的假设写进实现。

**转场：** 我们先看为什么这个项目适合作为案例，而不是随便选一个演示系统。

**素材来源：** `docs/00-项目基线与记录/产品与技术规范头脑风暴过程.md`。

### `P02` 为什么用这个项目做案例

**页面目的：** 说明企业售后工单系统虽然是 MVP，但具备真实项目的关键复杂度。

**页面可见文本建议：**

```text
一个小 MVP，足够观察完整工程过程

本项目不是“待办清单”式演示系统，而是包含：

1. 三类角色
   客户、客服、管理员的能力不同

2. 权限边界
   客户只能看自己的工单
   客服只能处理分配给自己的工单

3. 状态流转
   待分配 -> 处理中 -> 已解决 -> 已关闭

4. 数据持久化
   MVP 使用 JSON 文件，但后续要能替换真实数据库

5. 验收证据
   权限、安全、状态、持久化都必须可验证
```

**画面结构：**

- 左侧放项目卡片：`企业售后工单系统 MVP`。
- 右侧放五个观察点，每个观察点包含一个标题和一行解释。

**讲解重点：**

这个项目不是大型系统，但已经包含客户、客服、管理员三类角色；客户只能看自己的工单，客服只能处理分配给自己的工单，管理员能配置和分配；还有状态流、留言、操作记录、密码安全、JSON 持久化和未来数据库替换。它足够小，便于课堂讲完；也足够真实，能体现工程纪律。

**转场：** 在讲这个项目之前，我们要先划清课程边界：本课到底讲什么，不讲什么。

**素材来源：** `docs/00-项目基线与记录/企业售后工单系统-产品与技术规范-v1.0.md`。

```text
本课复盘过程，不推销工具

本课聚焦：
```

**画面结构：**

- 左侧为“本课聚焦”，蓝色。
- 右侧为“本课不展开”，浅灰并用橙色边线提示边界。

**讲解重点：**

课程关注的是“如何把软件想法变成可交付工程”，而不是展示某个工具有多强。Agent 能帮助推进很多材料，但课程要让学员理解每个产物为什么出现，以及它和后续开发、测试、验收的关系。

**转场：** 明确边界后，我们用一条主线贯穿全课：从不确定到可验证。

### `P04` 一条主线：从不确定到可验证

**页面目的：** 建立全课叙事框架，让学员知道后续 40 页按什么逻辑展开。

**页面可见文本建议：**

```text
好的协作，是逐步减少不确定性

模糊想法

选择题澄清
  服务谁？做什么？不做什么？如何流转？

产品基线

测试验收
```

## 6. 项目起点与需求澄清

### `P05` 项目起点：空目录和一句话

**页面目的：** 还原项目真实起点，说明从零开始时更需要澄清，而不是更可以省略澄清。

**页面可见文本建议：**

```text
起点不是代码，而是一个想法

项目初始状态：

- 工作目录为空
- 没有已有代码
- 没有现成文档
- 没有既定技术约束

原始想法：

“开发一个企业工单系统，先实现 MVP，再迭代开发。”

第一步不是生成代码，而是把隐藏选择问出来。
```

**可见示例：拟真会话片段**

```text
用户：
我有一个想法想和你讨论。
开发一个企业工单系统，先实现 MVP，再迭代开发。

协作要求：
一次问一个问题，帮助我完善这个想法。
```

**画面结构：**

- 左侧为空目录示意。
- 右侧为“原始想法”便签。
- 底部用箭头指向“第一步：澄清问题”。

**讲解重点：**

越是没有历史包袱，越不能直接让 Agent 生成代码。因为此时所有关键决策都还未确定，直接生成会把默认假设写入系统。正确做法是先通过问题把隐藏选择显性化。

**转场：** 第一个要问的问题，不是用什么框架，而是这个工单系统到底服务什么场景。

**素材来源：** `docs/00-项目基线与记录/产品与技术规范头脑风暴过程.md`。

### `P06` 第一类问题：先选业务场景

**页面目的：** 展示项目澄清从业务场景开始，而不是从功能清单或技术栈开始。

**页面可见文本建议：**

```text
第一个收敛：这是哪一种工单？

候选场景：

A. IT 服务台
   账号、设备、网络、软件安装、故障报修

B. 企业综合服务台
   IT、行政、人事、财务等统一受理

C. 客户支持 / 售后
   外部客户提交问题，内部处理与跟踪

D. 生产 / 运维事件
   系统告警、故障、变更、事件响应

确认结果：C. 客户支持 / 售后
```

**可见示例：为什么这个选择重要**

| 场景 | 典型角色 | 流程重点 |
| --- | --- | --- |
| IT 服务台 | 内部员工、IT 支持 | 内部服务请求与资产问题 |
| 客户售后 | 外部客户、客服、管理员 | 客户提交、内部分配、处理闭环 |
| 运维事件 | 值班、研发、运维 | 告警响应、影响控制、复盘 |

**画面结构：**

- 四个选项卡片横向排列。
- 已选的“客户支持/售后”用绿色对勾标识。
- 下方放小表格说明不同场景差异。

**讲解重点：**

同样叫“工单系统”，不同场景会导致角色、权限、流程和数据完全不同。第一问选错，后续所有文档都会偏。

**转场：** 场景定下来以后，第二件事是确定谁在系统里做什么。

**素材来源：** 头脑风暴过程第一问。

### `P07` 第二类问题：谁使用系统

**页面目的：** 展示角色定义如何影响权限、页面和接口。

**页面可见文本建议：**

```text
角色决定权限，也决定页面

客户
- 开放注册
- 登录客户门户
- 创建工单
- 查看本人创建的工单
- 在未关闭工单中追加公开留言

客服
- 由管理员创建账号
- 查看全部工单
- 按状态筛选
- 只处理分配给自己的工单

管理员
- 初始化后进入后台
- 维护问题分类
- 创建客服账号
- 启用 / 禁用客户账号
- 分配或重新分配工单
- 处理任意工单
```

**可见示例：权限边界一句话**

```text
客户：只能访问本人资源。
客服：能看全部工单，但只能处理本人负责工单。
管理员：拥有配置、分配和任意工单处理权限。
```

**画面结构：**

- 三列角色卡片。
- 每列上方是角色图标，下方是创建方式、数据范围和主要操作。

**讲解重点：**

角色不是权限矩阵里的最后一张表，而是从需求澄清阶段就要定。客户是外部个人用户，客服由管理员创建，管理员通过初始化产生。角色的创建方式、数据访问范围和处理权限，会继续影响注册、登录、页面导航、API 授权和测试用例。

**转场：** 有了角色，还要确认 MVP 的正范围：为了证明闭环，首期到底需要哪些能力。

**素材来源：** 头脑风暴过程第三、第四、十七、二十五、二十六、二十七问。

### `P08` 第三类问题：MVP 做什么

**页面目的：** 说明 MVP 正范围应围绕业务闭环，而不是堆积功能。

**页面可见文本建议：**

```text
MVP 只保留闭环所需能力

客户侧：
1. 开放注册
2. 基础登录 / 退出
3. 创建工单
4. 查看我的工单
5. 在工单详情中公开留言

管理员侧：
1. 初始化管理员
2. 维护问题分类
3. 创建客服账号
4. 启用 / 禁用客户账号
5. 手动分配 / 重新分配工单
6. 可直接处理任意工单

客服侧：
1. 查看全部工单
2. 按状态筛选
3. 处理分配给自己的工单
4. 回复并推进状态直至关闭
```

**可见示例：MVP 业务闭环**

```text
客户提交问题
  -> 管理员分配客服
  -> 客服公开回复并处理
  -> 工单进入已解决
  -> 内部人员关闭工单
  -> 客户查看处理结果
```

**画面结构：**

- 中间放“售后工单闭环”流程。
- 左右分别放客户侧、内部侧能力清单。

**讲解重点：**

MVP 不是功能越少越好，而是用最少能力验证核心价值。客户能提交问题，管理员能组织受理，客服能处理回复，客户能查看进展，数据能保留，这就能证明售后工单闭环成立。

**转场：** 只说做什么还不够，项目更容易失控的地方是没有说清暂不做什么。

**素材来源：** 产品与技术规范第 2、3、7 节；PRD 第 3、5、6、11 节。

### `P09` 第四类问题：MVP 暂不做什么

**页面目的：** 强调非范围清单对项目控制的价值。

**页面可见文本建议：**

```text
非范围清单，比功能清单更重要

MVP 暂不包含：

- 附件上传
- 邮件、短信或第三方通知
- SLA、超时提醒和升级流程
- 统计报表和管理看板
- 高级搜索、多条件筛选、批量操作、导出
- 客户确认关闭
- 工单取消、退回、重开、自动关闭
- 密码找回、多因素认证、验证码、登录失败锁定
- 真实关系型数据库迁移和正式生产部署

不是永远不做，而是不进入 MVP。
```

**可见示例：为什么要写非范围**

| 模糊说法 | 容易产生的问题 | 明确边界 |
| --- | --- | --- |
| “后面可能要通知” | 开发时顺手预留通知入口 | MVP 不建通知页面、接口和字段 |
| “以后可能要统计” | 首期加入报表数据结构 | 迭代 1 再做管理看板 |
| “客户最好能确认关闭” | 状态流变复杂 | MVP 由内部处理人员关闭 |

**画面结构：**

- 主体为橙色边界框，列出非范围项。
- 右侧放“小心范围膨胀”的提示。

**讲解重点：**

项目范围失控往往不是因为团队不知道要做什么，而是每个人都默认“这个也顺手加一下”。本项目明确首期不做附件、通知、SLA、报表、搜索、客户确认关闭和重开，后续迭代再安排管理看板、客服效率和正式运行基础。

**转场：** 范围收住以后，下一步要把核心流程定下来，尤其是状态流转。

**素材来源：** 产品与技术规范第 2、8 节；MVP 实施计划第 3.2 节。

### `P10` 第五类问题：流程如何闭环

**页面目的：** 展示状态流程是连接产品、权限、数据和测试的核心规则。

**页面可见文本建议：**

```text
状态流程越简单，规则越要清楚

MVP 状态流：

待分配
  客户已提交，等待管理员分配或介入

处理中
  内部人员已经开始处理

```

**可见示例：状态规则**

| 规则 | MVP 约定 |
| --- | --- |
| 能否跳级 | 不允许 `待分配 -> 已解决` |

**讲解重点：**

**转场：** 业务边界和流程定下来以后，才进入技术方向的确认。

**素材来源：** 产品与技术规范第 3 节；SRS 第 10 节。

### `P11` 第六类问题：技术方向如何确认

### `P15` SRS：把产品语言变成可开发规则

**页面目的：** 说明 SRS 如何把产品需求变成可实现、可测试的系统行为。

**页面可见文本建议：**

```text
SRS 把“能做”变成“必须如何做”

PRD 说：
客户应能创建工单。

SRS 必须继续回答：

- 谁能创建？
  仅已登录且启用的客户

- 创建什么？
  标题、问题描述、问题分类

- 字段有什么规则？
  标题 1-100 字，描述 1-4000 字

- 分类如何校验？
  必须引用当前启用的问题分类

- 创建后状态是什么？
  待分配，负责人为空

- 创建后能否修改？

```

**可见示例：SRS 需求片段**

| 编号 | 需求 |
| --- | --- |
| `SRS-TKT-001` | 仅已登录且启用的客户能够创建工单。 |
| `SRS-TKT-003` | 系统在保存工单前必须再次验证所选分类仍为有效状态。 |
| `SRS-TKT-004` | 创建成功时，系统应将工单状态设置为“待分配”，负责人设置为空。 |
| `SRS-TKT-005` | 工单创建后，MVP 不允许客户编辑标题、问题描述或分类。 |

**画面结构：**

- 左侧放 PRD 原始需求句。
- 右侧拆解为 SRS 问题与规则。
- 底部放编号需求表。

**讲解重点：**

SRS 的价值在于消除实现时的隐藏决策。系统必须知道输入字段、长度、唯一性、状态、权限、异常反馈、非功能要求。比如客户账号禁用后是否立刻失效、关闭后能否留言、分类停用后历史工单如何展示，这些都不能等开发时临时决定。

**转场：** SRS 说清系统行为后，还需要回答用户在哪些页面完成这些行为。

### `P17` 一个规则如何贯穿多份文档

**页面目的：** 用“客户不展示负责人信息”说明需求追踪和跨文档一致性。

**页面可见文本建议：**

```text
重要规则不能只写一次

示例规则：
客户不展示具体客服账号或负责人名称。

为什么重要？
客户需要看到处理进展，

```

**可见示例：跨文档追踪链**

| 文档 | 规则表达 |
| --- | --- |
| SRS | 客户只能访问允许的数据范围，不返回受保护内容。 |
| 页面与交互说明 | 客户详情页展示状态和公开留言，不展示具体负责人。 |
| API 接口设计 | 客户工单详情响应不包含负责人标识或名称。 |
| 测试与验收方案 | `TC-SEC-004` 检查客户详情响应不包含负责人和内部记录。 |
| 实施计划 | `M5` 复核响应字段与权限拒绝行为。 |

**画面结构：**

- 中间放示例规则。
- 周围用五个文档节点围绕，形成追踪链。
- 最后节点用绿色“验证”标记。

**讲解重点：**

一个关键规则如果只在某处提到，开发时很容易遗漏。客户不看负责人这件事，要在 SRS 写成数据边界，在页面说明中写成不展示，在 API 设计中写成响应不返回字段，在测试方案中写成检查项，在实施计划中写成安全复核。

### `P29` 质量门禁：哪些问题不能带过

**页面目的：** 明确 MVP 可以功能少，但不能降低核心质量底线。

**页面可见文本建议：**

```text
有些缺陷不能靠“后续优化”处理

MVP 验收红线：

- 越权访问
  客户看到他人工单，或客服处理非负责工单

- 敏感数据泄露
  客户看到负责人信息，响应包含密码哈希或会话摘要

- 明文密码保存
  数据文件中出现用户明文密码

- 状态违规写入
  跳级、回退、关闭后继续留言或重新分配

- 数据损坏或部分提交
  状态变了但操作记录没保存

- 业务闭环不可完成
  无法完成建单、分配、处理、关闭、查看
```

**可见示例：缺陷处理原则**

| 缺陷等级 | 示例 | 验收处理 |
| --- | --- | --- |
| 严重 | 越权读取、明文密码、JSON 损坏 | 必须修复并复测 |
| 高 | 状态可跳级、退出后仍可访问 | 必须修复并复测 |
| 中 | 重要交互反馈缺失 | 原则上修复，未修复需明确接受 |
| 低 | 不影响流程的展示问题 | 可记录后验收 |

- 上方列 MVP 红线。
- 下方放缺陷等级表。

**讲解重点：**

MVP 的范围可以小，但基本安全和业务一致性不能降级。客户看到他人工单、密码明文保存、关闭后仍能写入、JSON 损坏，这些都不是“后续优化”，而是验收阻断。

**转场：** 现在需求、设计和验收都清楚了，最后要回答开发按什么顺序做。

**素材来源：** 测试与验收方案第 7、14 节；实施计划第 12、13 节。

```text
  注册登录、建单、客户列表、详情、留言

M4 内部处理闭环
  内部列表、分配、留言、状态推进、操作记录

M5 安全、交互与完整联调
  字段隔离、XSS 文本展示、确认提示、浏览器验证

M6 验收准备与交付
  全量测试、端到端验收、运行说明、验收记录
```

**画面结构：**

- 横向或纵向里程碑时间线。
- 每个里程碑包含“阶段名 + 一句产出”。

**讲解重点：**

实施计划依据已经确认的产品、需求、设计和测试基线，安排开发顺序。先建工程骨架，再完成数据、仓储、安全和会话基础；然后做管理配置、客户工单、内部处理闭环；最后做安全交互联调和验收交付。

**转场：** 为什么这个顺序不是先做登录页、建单页、工单列表页？这就涉及任务拆解逻辑。

**素材来源：** MVP 实施计划第 6、7 节。

### `P31` 为什么不直接按页面开写

**页面目的：** 解释“技术基础先行、业务闭环主导、代码分层落地”的混合拆解方式。

**页面可见文本建议：**

```text
页面不是最小风险单元

如果直接按页面开写：

登录页
  可能临时处理会话

新建工单页
  可能临时处理分类校验

内部详情页
  可能临时处理权限和状态按钮

管理页面
  可能临时处理账号和审计记录

最后集中补权限、存储和测试
  返工风险高
```

**可见示例：推荐拆解方式**

```text
先固化共同基础
  数据模型、仓储协议、安全、会话、管理员初始化

再纵向完成业务闭环
  管理配置 -> 客户建单 -> 内部分配处理 -> 关闭

最后做系统复核
  字段隔离、错误反馈、浏览器联调、验收证据
```

**画面结构：**

- 左侧橙色“按页面开写”的风险路径。
- 右侧蓝色“基础先行 + 纵向闭环”的推荐路径。

**讲解重点：**

如果先按页面开写，会话、权限、JSON 原子写入、字段隔离都可能被各页面临时处理，后期统一困难。当前拆解方式先消除全局基础风险，再按业务闭环交付可验证增量。

**转场：** 在这些基础风险中，有一部分特别适合测试先行。

**素材来源：** 总体思路第 3 节。

### `P32` 哪些部分适合测试驱动

**页面目的：** 说明 TDD 在本项目中的选择性应用，而不是全项目教条化。

**页面可见文本建议：**

```text
高风险规则，适合先写测试

适合测试先行：

- 状态转换
- 权限判断
- 资源归属校验
- 会话失效
- 敏感字段隔离
- JSON 原子写入

不必强制严格 TDD：

- 普通页面排版
- CSS 视觉细节
- 一般提示文案微调
```

**可见示例：状态转换测试**

| 当前状态 | 下一状态 | 预期 |
| --- | --- | --- |
| 待分配 | 处理中 | 允许 |
| 处理中 | 已解决 | 允许 |
| 已解决 | 已关闭 | 允许 |
| 待分配 | 已解决 | 拒绝 |
| 已解决 | 处理中 | 拒绝 |
| 已关闭 | 任意状态 | 拒绝 |

**可见示例：字段隔离测试**

```text
客户工单详情响应不得包含：
- assignee_user_id
- assignee.username
- audit_logs
- password_hash
- token_hash
```

**画面结构：**

- 左侧列适合 TDD 的模块。
- 右侧放状态转换测试表和字段隔离检查。

**讲解重点：**

页面布局和普通样式不必严格 TDD，但状态、权限、会话、数据一致性和敏感字段隔离是高风险规则，适合先写允许与禁止路径测试，再实现代码。

**转场：** 在真正开工前，还要有一个检查清单，确认基线已经准备好。

**素材来源：** 总体思路第 2 节；测试与验收方案第 5、8-10 节。

### `P33` 开发启动前检查清单

```text
不是文档写完了，而是条件满足了

开发启动前检查：

[已完成] 产品与技术规范已经确认
[已完成] PRD、SRS 和页面交互说明已经确认
[已完成] 技术设计、数据模型和 API 设计已经确认
[已完成] 需求与设计一致性审查通过
[已完成] 测试与验收方案已经确认
[待确认] MVP 实施计划最终确认
[待启动] 代码工程初始化
```

**可见提示：**

```text
当前项目状态：
项目尚未开始代码开发，

```

**画面结构：**

**讲解重点：**

项目没有写代码，不代表没有开发过程；它已经完成了开发前很关键的一段工作。当前正在形成开发前实施计划，计划确认后才进入工程初始化与编码。

## 11. Agent 协作复盘

### `P34` Agent 在这个过程中做了什么

**页面目的：** 总结 Agent 在项目构建过程中的实际作用。

**页面可见文本建议：**

```text
Agent 的价值，是加快分析和产物衔接

在本项目中，Agent 主要做了：

1. 提出澄清问题
   把模糊想法拆成可选择的决策点

2. 整理决策记录
   保存头脑风暴过程和确认背景

3. 生成规范文档
   产品与技术规范、PRD、SRS、页面说明

4. 展开设计文档
   技术设计、数据模型、API 接口

5. 建立追踪关系
   需求与设计追踪矩阵、一致性审查

6. 编制测试与验收方案
   测试用例、E2E 场景、退出准则

7. 拆解实施计划
   M0-M6 里程碑、任务、质量门禁
```

**画面结构：**

- 中央放 Agent 协作节点。
- 左侧是用户输入与确认。
- 右侧是项目产物链。

**讲解重点：**

Agent 在本项目中承担的是分析、整理、生成、检查和计划拆解。它没有替用户决定客户售后场景，也没有擅自扩大 MVP。每次关键决策都需要用户确认，确认后 Agent 再把它转成文档或计划。

**转场：** 因此，必须同时讲清人的责任位置。

**素材来源：** 头脑风暴过程全篇；各基线文档确认记录。

### `P35` 人在这个过程中负责什么

**页面目的：** 强调人在方向、范围、确认和责任上的不可替代性。

**页面可见文本建议：**

```text
Agent 可以推进工作，人必须批准方向

人在本项目中负责：

- 选择业务场景
  确认 MVP 是客户支持 / 售后

- 确认客户与账号模式
  外部个人客户，客户门户开放注册

- 确认处理组织
  客服 + 管理员，管理员手动分配

- 确认范围取舍
  首期不做附件、通知、SLA、报表和高级搜索

- 确认技术约束
  原生前端、FastAPI、JSON 持久化、可替换仓储

- 批准基线
  产品与技术规范、PRD、SRS、设计、测试方案

- 判断是否进入下一阶段
  实施计划确认后才开始编码
```

**画面结构：**

- 上方是人的判断与批准节点。
- 下方是 Agent 分析、生成、检查节点。
- 中间用双向箭头表示协作。

**讲解重点：**

本项目中，用户决定选择客户支持/售后，决定客户门户和个人客户账号，决定 JSON 存储但要可替换数据库，决定 MVP 后的迭代路线。Agent 可以给出选项和解释，但业务取舍、批准和最终责任仍属于人。

**转场：** 这个项目也暴露了一些协作教训，这些教训比顺利产出更值得讲给学员。

**素材来源：** 头脑风暴过程中的选择题确认；产品与技术规范确认记录。

### `P36` 这个案例暴露的协作教训

**页面目的：** 用真实过程提炼可迁移的协作纪律。

**页面可见文本建议：**

```text
Agent 协作也需要过程纪律

教训 1：确认后要落盘
  聊天中确认过，不等于项目基线已经形成。

教训 2：建议措辞要定稿
  已确认设计不能继续写成“建议”“可考虑”。

教训 3：发现项要关闭
  发现歧义后，要修订文档并记录关闭结论。

教训 4：非范围不要偷做
  附件、通知、报表等后续能力不能悄悄塞进 MVP。

教训 5：测试标准不能后补
  验收口径应在编码前确认，而不是开发后临时解释。
```

**可见示例：本项目中的小偏差**

| 现象 | 修正 |
| --- | --- |
| 规范确认后未立即保存 | 补充落盘为正式基线文档 |
| 页面负责人展示存在歧义 | 修订为客户详情不展示具体负责人 |
| 技术设计仍有建议措辞 | 改为确定性设计约束 |

**画面结构：**

- 五张经验卡片。
- 下方放真实小偏差与修正表。

**讲解重点：**

AI Agent 时代并不意味着工程纪律变少。相反，因为产出速度更快，基线、版本、确认、审查和验收标准更重要。

**转场：** 把这些经验抽象出来，就能形成一套可复用的协作模式。

**素材来源：** 头脑风暴过程“规范形成与确认”；一致性审查记录第 7、9 节。

### `P37` 可复用协作模式

**页面目的：** 把本案例抽象为学员可复用的工作步骤。

**页面可见文本建议：**

```text
用同一套节奏处理下一个项目

1. 先问清场景
   这个系统到底服务什么业务？

2. 确认 MVP
   首期必须证明哪条业务闭环？

3. 写清非范围
   哪些能力后续再做？

4. 固化产品基线
   已确认内容落盘，成为后续依据。

5. 拆成需求与交互
   PRD、SRS、页面说明。

6. 展开系统设计
   技术设计、数据模型、API。

7. 先定测试验收
   用例、E2E、退出准则。

8. 再排实施计划
   按风险和依赖组织开发顺序。
```

**可见总结句：**

```text
每一步都有确认，每一步都有产物。
```

**画面结构：**

- 八级阶梯流程图。
- 每一步配一个产物图标。

**讲解重点：**

学员不一定要复制本项目全部文档名称，但可以复制节奏：先澄清，再确认，再落盘，再追踪，再验证，再实施。Agent 最适合在这个节奏中帮助生成结构化产物和检查一致性。

**转场：** 课程最后，我们把一句话总结出来：这不是让项目变慢，而是减少返工。

**素材来源：** 本项目完整文档链路。

## 12. 收束与讨论

### `P38` 总结：这不是慢，而是少返工

**页面目的：** 形成课程核心结论。

**页面可见文本建议：**

```text
规格清楚，Agent 才能跑得稳

前期基线不是为了拖慢开发，
而是为了让后续开发：

- 范围有边界
- 实现有依据
- 测试有证据
- 变更可控制
- 验收可判断

Agent 提高执行速度，
清晰规格决定速度是否能变成可靠交付。
```

**可见对比：**

| 没有基线的快 | 有基线的快 |
| --- | --- |
| 快速生成，但假设不清 | 快速执行，边界清楚 |
| 页面能跑，但权限不稳 | 流程可用，规则可测 |
| 需求变了难追踪 | 变更影响可定位 |
| 验收时再争论 | 开发前已定义通过标准 |

**画面结构：**

- 中央大观点。
- 下方左右对比表。

**讲解重点：**

前期基线不是为了拖慢开发，而是为了让后续开发少返工、少争议、少隐藏风险。Agent 提高速度后，更需要明确输入和验收，否则只是更快地产生不确定产物。

**转场：** 为了让大家把方法带走，我们安排一个简单练习。

### `P39` 学员练习：拿自己的想法走一遍

**页面目的：** 引导课堂互动，帮助学员迁移方法。

**页面可见文本建议：**

```text
练习：把一句想法变成 MVP 边界

请用你自己的一个软件想法，完成：

1. 写一句原始想法
   例如：做一个内部知识库系统。

2. 设计 5 个澄清问题
   服务谁？入口是什么？角色有哪些？首期流程是什么？不做什么？

3. 写 3 项 MVP 必做
   例如：创建知识条目、搜索条目、管理员审核。

4. 写 5 项 MVP 暂不做
   例如：积分、推荐、外部分享、复杂权限、移动端。

5. 定义 1 条端到端验收路径
   例如：管理员建分类 -> 用户提交知识 -> 管理员审核 -> 其他用户搜索查看。
```

**可见练习模板：**

| 项目 | 填写 |
| --- | --- |
| 原始想法 |  |
| 目标用户 |  |
| 5 个澄清问题 |  |
| MVP 必做 3 项 |  |
| MVP 暂不做 5 项 |  |
| 端到端验收路径 |  |

**画面结构：**

- 左侧为练习步骤。
- 右侧为可填写模板。
- 底部放时间建议：`个人 5 分钟，小组讨论 8 分钟`。

**讲解重点：**

讲师可以让学员两三人一组，选择一个真实想法练习。重点不是写得完整，而是体验“先问问题、再定边界”的节奏。

**转场：** 最后用几个讨论问题收束，看看每个团队最容易跳过哪一步。

### `P40` 讨论题与结束页

**页面目的：** 收束课程并引发学员反思。

**页面可见文本建议：**

```text
讨论：你的团队最容易跳过哪一步？

1. 需求澄清
   一开始就默认大家理解一致

2. 非范围确认
   只写要做什么，不写暂不做什么

3. 设计追踪
   需求、页面、数据、API 之间缺少映射

4. 测试前置
   代码写完后才讨论怎样算完成

5. 变更确认
   开发过程中悄悄扩大范围

结束语：
Agent 加快执行，人负责判断；
基线越清楚，协作越可靠。
```

**画面结构：**

- 上方五个讨论问题。
- 下方一句结束语。
- 右下角放“谢谢 / Q&A”。

**讲解重点：**

讲师可根据学员岗位引导讨论：产品经理通常对非范围最有感，项目经理关注实施计划和变更确认，研发管理者关注设计追踪和测试前置。

**素材来源：** 课程整体结论。
