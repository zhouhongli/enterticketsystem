## 1. 文档目的

本文档定义一套可进一步转化为演示文稿（PPT）的培训内容框架，帮助产品、项目与研发管理人员理解软件开发模式正在发生的变化。培训并非介绍某一款 AI 产品，也不以推动具体工具采购或团队流程改造为目标，而是回答一个管理视角下的核心问题：

> 当 AI Agent 能够在项目上下文中参与分析、执行和验证时，软件开发中的协作方式、管理关注点和人的责任发生了什么变化？

培训将回顾瀑布/V 模型、敏捷与 DevOps 的主要组织思路，并与当前已经能够实践的 AI Agent 协作模式进行对照。内容覆盖从需求形成至运行反馈的完整软件生命周期，同时保持平衡立场：说明 Agent 带来的效率与协作机会，也说明结果正确性、质量、安全、合规与最终责任仍需由人和组织保证。

## 3. 核心概念与基本立场

### 3.1 三类开发模式的课程定义

| 模式 | 本课程中的定义 | 主要价值 |
| --- | --- | --- |
| 瀑布/V 模型 | 通过阶段化需求、设计、实现、验证与交付，以及相互对应的测试活动，管理大型协作中的范围、质量与追踪。 | 建立计划、文档、评审、职责和验证关系。 |
| 敏捷与 DevOps | 通过迭代开发、持续反馈、自动化交付与运行反馈，缩短从需求到价值验证的周期。 | 提高响应速度、持续集成和运行反馈能力。 |
| AI Agent 协作模式 | 人在明确目标、上下文、授权和验证要求下，将分析或执行活动委派给能够读取项目资料、生成或修改产物、运行工具并汇报结果的 Agent。 | 提高信息处理、执行及验证反馈的密度。 |

### 3.2 对 Agent 的当前可落地定义

本课程中的 `AI Agent` 不是未来设想中的全自动软件组织，也不仅是聊天问答或单行代码补全工具。它指的是当前已经可以实践的一类协作执行者：在获得工作上下文与操作权限后，Agent 可以帮助阅读资料、澄清需求、比较方案、拆解任务、编辑代码或文档、运行检查和测试、分析结果并向人报告。

其前提是人仍然承担以下职责：

- 明确要解决的问题、成功标准和禁止越过的边界。
- 决定范围取舍、业务优先级、架构承诺和重大变更。
- 审查重要产物与验证证据，批准发布或高风险操作。
- 对安全、合规、客户影响和最终交付结果负责。

### 3.3 基本立场

课程不将三类模式描述为简单的替代顺序。瀑布/V 模型所强调的可追踪基线和验证责任、敏捷与 DevOps 所强调的快速反馈和持续交付，在 Agent 协作模式中依然有价值。Agent 改变的是可委派工作的粒度与反馈速度，而不是取消工程纪律或人的责任。

## 4. 课程叙事主线

本培训以“软件开发中的协作单位如何变化”为主线：

```text
阶段与交付件
    瀑布 / V 模型：通过正式阶段、文档、评审和测试对应关系控制复杂度
        |
        v
迭代与持续反馈
    敏捷 / DevOps：通过较小批次、持续集成和运行反馈缩短价值验证周期
        |
        v
上下文内的可委派活动
    AI Agent 协作：将澄清、检索、整理、实现、验证和同步等具体活动
    交给受约束的 Agent 加速执行，并由人审查和承担责任
```

课程需要帮助听众认识到：新的变化并不只表现为“代码生成更快”。更深层的变化在于，过去需要不同人员跨会议、文档和排期完成的部分信息处理及执行工作，现在可能在一次受控协作中快速完成；与此同时，目标模糊、约束遗漏或审查不足也可能更快地转化为错误结果。

## 5. 总体比较框架

| 比较维度 | 瀑布/V 模型 | 敏捷与 DevOps | AI Agent 协作模式 |
| --- | --- | --- | --- |
| 主要协作单位 | 阶段、正式文档、评审和交接 | 迭代、用户故事、流水线和反馈闭环 | 带上下文、约束与验证要求的可委派任务 |
| 需求处理 | 尽量提前明确并基线化 | 按迭代细化并随反馈调整 | 人确定意图与取舍，Agent 辅助提问、整理、核对和更新产物 |
| 设计处理 | 专门设计阶段与评审 | 适度预先设计并持续演进 | Agent 可比较选项和检查一致性，人批准关键架构承诺 |
| 实现活动 | 按设计由开发人员编码 | 迭代实现并持续集成 | Agent 可受控修改代码、文档或配置，并报告变更与验证 |
| 验证方式 | 后续或配套的测试阶段 | 自动化测试和持续验证 | Agent 可运行验证、定位失败并补充证据，但通过判定仍需可信门禁与人工审查 |
| 反馈节奏 | 较长阶段后集中反馈 | 短迭代和运行期持续反馈 | 单个任务内即可发生分析、修改、验证、修订的密集循环 |
| 主要管理风险 | 变更成本高、反馈较迟 | 范围漂移、短期交付压力 | 错误快速扩散、授权越界、产物未经审查即被采信 |
| 人的关键责任 | 制定并审批基线、阶段验收 | 决定价值顺序、审查增量和运行反馈 | 提供目标与约束、控制授权、审查证据并对结果负责 |

## 6. 全生命周期对照内容

| 生命周期环节 | 传统阶段式关注点 | 敏捷/DevOps 中的变化 | AI Agent 协作模式中的变化 | 管理者应关注的问题 |
| --- | --- | --- | --- | --- |
| 构思与需求 | 访谈、调研、整理需求文档并正式确认 | 通过用户故事、原型和反馈持续澄清 | Agent 可协助连续提问、归纳冲突、整理需求基线和检查遗漏 | 业务目标和范围取舍是否由责任人确认；Agent 是否把假设写成结论 |
| 规划与设计 | 完成架构、数据、接口和计划评审后实施 | 设计随迭代演进，保留必要技术治理 | Agent 可检索既有约束、比较方案、拆解计划并检查跨产物一致性 | 关键架构承诺、数据与安全边界是否得到人工审批 |
| 编码实现 | 开发人员依据设计编写代码并交接测试 | 小批次实现、合并审查与持续集成 | Agent 可在工作区读取上下文、进行受控编辑、说明修改并处理反馈 | 任务权限是否适当；变更是否可读、可审查、没有越界扩展范围 |
| 测试验证 | 测试阶段依据规格执行验证 | 自动化回归与流水线不断反馈 | Agent 可运行测试、分析失败、修复缺陷、生成验证摘要 | 验证是否真实执行；高风险规则是否依赖独立证据而非 Agent 声称 |
| 发布交付 | 按发布计划审核、部署和验收 | 自动化部署、渐进发布和快速回滚 | Agent 可辅助准备发布说明、核对检查项或分析发布失败 | 上线批准、凭证权限、回滚决定不得无控制地委派 |
| 运行维护 | 人工值守、问题处理、定期复盘 | 监控、告警、事件响应与反馈进入待办 | Agent 可归纳日志与反馈、提出问题定位方向或变更建议 | 隐私和敏感数据是否受保护；诊断建议是否经过复核 |
| 迭代反馈 | 后续版本重新进入需求和设计周期 | 依据真实反馈持续调整产品待办 | Agent 可聚合反馈、维护文档差异、辅助形成下一轮候选范围 | 数据解释、优先级和产品决策是否仍由人承担 |

### 7.2 第一章：为什么重新理解软件开发模式

**章节目标：** 建立讨论背景，将问题从“AI 能否生成代码”提升到“软件生产如何被重新组织”。

**内容要点：**

1. 软件开发模式始终在解决两个问题：复杂协作如何可控，价值反馈如何更快。
2. AI Agent 的出现使分析、执行、验证和文档同步可以在更小任务颗粒上连续发生。
3. 对管理人员而言，需要重新审视的不只是研发工时，而是职责、评审、证据与风险。

### 7.3 第二章：瀑布/V 模型到敏捷与 DevOps

**章节目标：** 使听众理解 Agent 模式并非从真空出现，已有模式分别沉淀了仍需保留的工程价值。

**内容要点：**

1. 瀑布模式通过阶段和交付件管理范围及分工；V 模型强化需求、设计与验证的对应关系。
2. 其限制是反馈可能较晚，变更成本较高，难以快速验证用户价值。
3. 敏捷以增量需求和短迭代获得更快反馈；DevOps 将构建、测试、发布及运行反馈进一步连接。
4. Agent 协作并不否定基线、测试、审查或持续反馈，而是使用这些基础约束更高频率的执行。

### 7.4 第三章：当前可落地的 AI Agent 协作开发

**章节目标：** 给出课程对 Agent 的严谨定义，说明当前实践中的协作边界，并以一页框架示例让概念落地。

**内容要点：**

1. 区分聊天问答、代码补全、自动化脚本与能够读取上下文、执行任务、运行验证并反馈结果的 Agent。
2. Agent 可承接的不是“最终责任”，而是受目标、范围、权限和验证要求约束的分析或执行活动。
3. 人应负责方向、约束、关键审批、证据审查和结果责任。
4. 公开实践已经呈现“先明确意图和规格，再计划、执行、验证与审查”的趋势。

#### `P09` 正文示例：已经出现的实践框架

本页只用于说明实践形态，不作为工具推荐、行业标准判断或选型比较。

| 框架 | 公开定位所体现的侧重点 | 本课程提取的共同启示 |
| --- | --- | --- |
| `Superpowers` | 面向 coding agents 的技能框架与软件开发方法，包含头脑风暴、设计确认、计划、测试驱动、代码审查与完成验证等流程。 | Agent 的速度需要由澄清、计划、测试与审查约束。 |
| `Spec Kit` | 面向 Spec-Driven Development 的开源工具包，强调先定义产品场景与预期结果，并通过规格、计划与任务支持实现。 | 规格不仅是交接文档，也可以成为 Agent 执行工作的结构化输入。 |
| `OpenSpec` | 面向 AI coding assistants 的轻量规格驱动方式，以变更提案、规格、设计和任务组织单次变化。 | Agent 协作需要让意图和变更保持可见、可讨论、可追踪。 |

**本页结论：**

```text
这些框架实现方式不同，但共同说明一种趋势：
Agent 开发正在从即时问答和零散生成，
走向“明确意图 -> 表达规格 -> 拆解任务 -> 受控执行 -> 验证审查 -> 保留变化记录”的工程协作。
```

### 7.5 第四章：全生命周期中的模式对照

**章节目标：** 将模式差异落实到各个研发管理环节，防止听众将 Agent 理解为单纯的编码能力。

**内容要点：**

1. 需求形成：Agent 可帮助提问、整理和一致性检查；价值判断与范围确认仍在人。
2. 规划设计：Agent 可辅助检索约束、推演方案和拆解任务；架构、安全和成本承诺仍需审批。
3. 编码测试：Agent 可受控修改、运行验证与分析失败；质量判定不能只依赖其自我报告。
4. 发布运维：Agent 可协助整理发布内容和诊断信息；上线批准、敏感访问和事故责任不可失控委派。
5. 反馈迭代：Agent 可汇总信息、同步产物；优先级和产品决策仍属于团队负责人。

| `P10` | 生命周期总览图 | 将需求到迭代反馈的全链路放在同一张图中。 |
| `P11` | 构思与需求：从记录需求到辅助澄清 | 人确认目标、价值和范围。 |
| `P12` | 规划与设计：从文档交接到上下文协作 | 关键约束和架构批准仍由人控制。 |
| `P13` | 编码与测试：从人工执行到受控任务委派 | 修改、验证和证据形成更密集。 |
| `P14` | 发布与运维：从流程执行到辅助诊断 | 权限、审批与运行风险边界。 |
| `P15` | 迭代反馈：从复盘材料到快速归纳 | Agent 加快信息闭环，不替代产品判断。 |

### 7.6 第五章：价值、局限与管理挑战

**章节目标：** 保持机会与风险的平衡表达，使管理人员形成判断能力，而不是简单的乐观或排斥态度。

**价值要点：**

| 价值变化 | 表现 |
| --- | --- |
| 信息处理更快 | 可以更快阅读上下文、比较材料、发现冲突并整理结构化输出。 |
| 活动衔接成本下降 | 需求、设计、任务、实现和验证之间的同步工作可被辅助完成。 |
| 可验证反馈更密集 | 在一个任务循环内完成修改、运行检查、分析失败和汇报证据。 |

**局限与风险要点：**

| 风险 | 说明 |
| --- | --- |
| 错误快速落实 | 不正确的需求假设或边界可能被快速生成到设计、代码和文档中。 |
| 虚假的完成感 | 产物生成很快，不代表测试执行、验收通过或业务目标成立。 |
| 上下文与权限风险 | 输入材料不足会造成偏差；权限过大可能带来敏感数据或不当修改风险。 |
| 责任模糊 | 若团队把 Agent 输出直接当作决策，可能弱化审批、审查和事故责任。 |

**建议页面：**

| 页面 | 标题 | 核心信息 |
| --- | --- | --- |
| `P16` | Agent 带来的三类价值 | 信息、衔接、反馈速度。 |
| `P17` | 速度不会自动变成质量 | 错误扩散、完成错觉和授权风险。 |
| `P18` | 新模式仍需要的控制原则 | 授权边界、人工审查、验证证据、敏感保护、变更追踪。 |

### 7.7 第六章：管理者应建立的新认识

**章节目标：** 用简洁结论收束课程，不展开具体导入实施蓝图。

**内容要点：**

1. 瀑布/V 模型提供的基线、职责与验证关系仍值得保留。
2. 敏捷与 DevOps 提供的短周期反馈和持续验证仍是提高交付质量的重要基础。
3. AI Agent 使可委派活动进一步细化，并显著提高执行与反馈密度。
4. 人的价值并未消失，而是更集中地体现在正确设定目标、提供约束、判断结果、控制风险和承担责任。
5. 判断一个团队是否真正进入 Agent 协作模式，不在于是否使用 AI，而在于能否以可审查、可验证、可追踪的方式让 Agent 参与工作。

**建议页面：**

| 页面 | 标题 | 核心信息 |
| --- | --- | --- |
| `P19` | 三种模式不是三选一 | 保留工程基础，吸收反馈机制，理解 Agent 新能力。 |
| `P20` | 管理者需要回答的五个问题 | 目标、上下文、授权、证据、责任。 |
| `P21` | 结语 | Agent 加速软件工作，人定义方向并对结果负责。 |
| `P22` | 参考资料 | 实践框架与必要术语参考链接。 |

### 8.3 表达风格

- 采用“对照”而非“替代”的表达，避免把传统方法表述为无价值旧方法。
- 采用“当前可落地”而非“完全自治”的表述，避免把未来设想当作现实能力。
- 采用管理语言说明活动、产物、责任和风险，减少具体工具操作细节。
- `Superpowers`、`Spec Kit` 和 `OpenSpec` 仅出现于一页实践示例及参考资料中。
- 当前企业售后工单系统不作为课程主线；如后续制作 PPT 需要，可仅作为补充示例另行确认。
