## 1. 制作定位

本演示文稿不是 AI 工具推介，也不预设“传统开发将被替代”的结论。它面向需要判断研发协作方式与治理边界的管理人员，以同一个轻量教学案例完整呈现三种模式，再进行横向比较：

1. `瀑布 / V 模型`：以阶段基线、评审与测试对应关系控制复杂度。
2. `敏捷 / DevOps`：以小批次迭代、自动验证和运行反馈缩短周期。
3. `AI Agent 协作`：以明确上下文、可委派任务、工具验证和人工审批提高反馈密度。

贯穿案例为“为业务系统新增客户反馈功能”，仅包含两个必要能力：

- 用户提交反馈内容。
- 管理员查看反馈列表。

案例只为教学表达服务，不关联当前企业售后工单系统范围，也不暗含状态流转、通知、统计、复杂权限或其他扩展需求。

### 2.3 页面设计原则

- 先给足事实与过程，再形成结论；不让主张悬空。
- 每种模式均具备定义、流程全景、拟真产物、质量责任与小结。
- 案例内容保持一致，使比较发生在开发方式而非需求差异上。
- `Spec-Driven Development` 按公开资料呈现共同趋势与不同实践，不表述为唯一标准。
- 页数不作为预先约束；当前脚本形成 `39` 页制作序列，可在正式制作审阅后合并或拆分。

## 10. 重点页面制作说明

### 10.1 三种“流程全景页”

`P07`、`P13`、`P19` 必须使用相同的三层结构和同一案例范围，使听众能够直接观察变化：

```text
参与角色       谁提出、谁实施、谁审查、谁批准
------------------------------------------------
工作流程       需求/规格 -> 执行 -> 验证 -> 交付
------------------------------------------------
形成产物       用于沟通、证明和批准的材料
```

三页的差异不依靠配色制造，而依靠流程节点、反馈回路和 Agent 活动范围真实呈现。

### 10.2 三组“拟真产物页”

拟真片段应看起来像工作中会阅读的材料，而不是抽象标签：

| 模式 | 页面 | 必须显示的材料 |
| --- | --- | --- |
| 瀑布 / V | `P08-P09` | 需求编号、设计条目、测试映射与通过证据 |
| 敏捷 / DevOps | `P14-P15` | 用户故事、验收标准、任务状态、流水线记录与反馈 |
| AI Agent | `P23-P24` | 上下文边界、确认规格、任务指令、修改摘要、验证与审批 |

### 10.3 Agent 纠偏页

`P29` 不只展示“存在风险”，必须展示一次闭环过程。错误项固定使用“擅自加入通知能力”，与 `P04` 设定的排除范围前后对应；重新验证必须包含范围检查，避免把纠偏仅表现为删去一项文字。

## 11. PPT 制作验收清单

| 验收项 | 判断标准 |
| --- | --- |
| 内容完整性 | 三种模式均含定义、全景、产物、质量责任和小结 |
| 案例一致性 | 全程只围绕提交反馈与查看列表展开 |
| 事实准确性 | 三个框架按公开资料表述，不声称统一替代方案 |
| 责任表达 | 每种模式均明确评审、验证与发布批准责任 |
| 信息可读性 | 普通正文为 `18pt`；拟真片段可读且不过度拥挤 |
| 视觉统一 | 白底、蓝主色、橙绿辅助，使用统一线性图标 |
| 讲授配合 | 页面顺序与独立讲师稿编号一致 |
