## 1. 讲授原则

本讲师稿负责扩展解释页面中的简洁信息，不要求将所有文字搬到 PPT 上。授课时应始终保持四项边界：

1. 讨论的是开发模式及管理判断，不是工具安装、操作或选型课程。
2. 三种模式先分别完整说明，再进行比较；不以“新旧淘汰”代替事实分析。
3. 客户反馈功能为教学案例，仅含用户提交反馈和管理员查看列表。
4. 对 `Spec Kit`、`OpenSpec`、`Superpowers` 的陈述以官方公开资料为依据，只说明实践取向，不推断统一标准。

## 2. 开场与比较基准

### `P01` 封面：开发模式正在改变什么

**讲解重点：**

今天讨论的不是某个模型能写多少代码，而是软件工作如何被组织。管理人员需要关心的是：需求怎样形成可执行任务，过程中留下什么证据，错误如何被发现，最终由谁批准与负责。课程会从阶段式开发、迭代交付讲到当前可落地的 Agent 协作。

**转场：** 先把课程最终希望大家带走的判断讲清楚。

### `P02` 课程结论预告：提效不等于责任转移

**讲解重点：**

Agent 可以协助读取资料、执行任务、修改产物和运行验证，因此执行循环可能明显加快。但执行速度不是责任主体，Agent 也不会自动知道组织真正允许什么、什么风险能够接受。目标、权限、审批与最终责任必须有明确的人承担，这也是后面所有比较的主线。

**转场：** 有了这个立场，还要限定本课程的内容范围。

### `P03` 本课程讲什么，不讲什么

**讲解重点：**

课程聚焦模式理解：每种模式如何工作，产生什么材料，以什么方式保障质量。框架名称会出现，是为了说明业界已经出现把 Agent 协作工程化的公开实践，不是推荐采购或教授命令。团队如何导入、如何制定企业制度，则应在建立共识后另行设计。

**转场：** 要进行有效比较，三种模式必须面对同一个问题。

### `P04` 贯穿案例：新增客户反馈功能

**讲解重点：**

案例刻意保持小而明确：用户能够提交一段反馈，管理员能够在列表中查看记录。它足以观察需求、流程、产物、测试和责任，同时避免复杂业务逻辑干扰模式比较。需要特别注意四个排除项，后续 Agent 纠偏案例会用到这一范围边界。

**转场：** 接下来用固定的七个观察问题贯穿全课。

### `P05` 比较坐标：看七个问题

**讲解重点：**

比较模式不能只比较“快不快”。同一功能在三种模式下，应分别观察需求表达、工作组织、参与角色、交付产物、质量证据、发布责任以及反馈方式。这样才能看出变化发生在哪些活动上，哪些工程纪律一直没有消失。

**转场：** 首先从最强调阶段基线与责任追踪的方式开始。

## 3. 瀑布 / V 模型讲解

### `P06` 模式定义：先建立基线，再逐阶段交付

**讲解重点：**

瀑布和 V 模型往往被简单概括为“慢”，但对管理而言，其核心价值是范围和验证关系清晰：需求得到确认后，设计、开发与测试以基线为依据推进；V 模型进一步强调测试活动应对应前序定义。这种方法适合需要明确批准和可追踪交付的工作。

**转场：** 把这一逻辑放进客户反馈功能，可以看到一条完整交付链。

### `P07` 流程全景：客户反馈功能如何被交付

**讲解重点：**

业务负责人首先确认只做提交与列表；需求分析形成正式条目；设计开发依据评审后的设计实施；测试人员以基线检查结果；发布负责人依据验收证据批准交付。这里每一步之间都存在交接和门禁，管理优势是责任位置容易说明，代价是变化需要重新穿过部分链路。

**转场：** 阶段控制不是抽象概念，它落在实际可阅读的文档中。

### `P08` 拟真产物一：需求与设计基线

**讲解重点：**

左侧需求条目明确两项能力与一项排除范围，右侧设计说明把范围转换为页面和必要数据表达。这里的重点不是示例字段本身，而是团队在实现之前有一份共同依据：哪些内容必须做，哪些内容暂不做，后续争议可以回到基线核对。

**转场：** 只有文档还不够，V 模型还要求每项承诺能够被验证。

### `P09` 拟真产物二：测试对应与追踪

**讲解重点：**

需求条目与测试用例一一映射，使团队可以回答“这项需求如何证明完成”。排除项同样需要核验，例如本次没有加入通知和状态处理。管理者在这里获得的是证据链：不是听到“功能做完了”，而是知道已确认范围分别经过什么检查。

**转场：** 证据链背后，还需要明确谁拥有批准权。

### `P10` 质量与责任：阶段门禁由谁把关

**讲解重点：**

阶段式交付中，质量保障不是测试环节单独负责：业务负责人确认需求范围，技术负责人审批方案，测试负责人提供验收证据，发布责任人决定是否交付。每道门都可以拒绝不满足条件的产物，因此责任清晰是这种模式的核心资产。

**转场：** 至此可以平衡地评价这种方式适合解决什么问题。

### `P11` 本模式小结：稳定范围的强控制方式

**讲解重点：**

当需求相对稳定、审查要求强、责任追踪重要时，阶段式方式有明显价值。但一旦用户反馈导致方向经常变化，重新确认、重新设计与重新验收的时间成本会变得突出。因此后续模式并不是否定基线，而是试图让反馈更早发生。

**转场：** 第二种模式把注意力从一次性阶段推进转向小批次反馈。

## 4. 敏捷 / DevOps 讲解

### `P12` 模式定义：以可工作的增量验证价值

**讲解重点：**

敏捷强调用可验证的增量尽早获取反馈，DevOps 则将构建、测试、发布和运行信息更紧密地连接起来。它们不是减少质量要求，而是把质量验证更频繁地嵌入交付循环。对管理者而言，核心变化是反馈更快，决策也需要更持续。

**转场：** 在相同案例中，流程会从长链路变为较短的迭代循环。

### `P13` 流程全景：小批次完成同一功能

**讲解重点：**

客户反馈功能可以作为一个可演示的小增量进入迭代：产品负责人确认故事与验收标准，团队开发并测试，流水线提供基本验证结果，演示获得业务反馈，具备批准条件后交付。反馈可以回到待办池，但不会自动扩张本轮已承诺范围。

**转场：** 迭代方式的沟通材料更轻量，但同样具体。

### `P14` 拟真产物一：故事、验收标准与任务板

**讲解重点：**

用户故事表达的是价值视角，验收标准把价值转换为可检查条件，任务板显示增量当前处在什么状态。与阶段式需求文档相比，它的颗粒度更适合短周期讨论。关键点仍然是可检查：用户提交后是否保存，管理员打开列表是否可看到记录。

**转场：** 当开发动作频繁发生，验证证据也需要更及时形成。

### `P15` 拟真产物二：流水线证据与发布反馈

**讲解重点：**

流水线记录把构建、自动测试和范围检查留成可复查证据；演示中产生的新建议，例如以后增加统计，不应被悄悄塞进当前交付，而应进入后续讨论。敏捷处理变化的能力，来自有秩序地接收变化，而不是随意扩大任务。

**转场：** 更频繁的反馈并不会取消批准和责任要求。

### `P16` 质量与责任：高频交付仍需门禁

**讲解重点：**

产品负责人确认验收标准与后续优先级，团队对实现和测试持续负责，自动化流水线提供高频证据，具备授权的人仍决定是否发布。自动化能帮助迅速发现部分问题，但不能替代业务判断、安全审查或上线责任。

**转场：** 这一模式的收获和要求可以归纳为反馈距离的变化。

### `P17` 本模式小结：缩短假设到反馈的距离

**讲解重点：**

敏捷与 DevOps 适合需要更早验证用户价值、持续调整方向的工作。其有效前提是待办管理、自动测试和发布控制都可信，否则更快的迭代也可能更快积累问题。这一点为理解 Agent 协作提供了重要基础：加速必须建立在验证之上。

**转场：** 第三种模式进一步改变了执行活动可以被委派的颗粒度。

## 5. AI Agent 协作讲解

### `P18` 当前可落地定义：Agent 进入工作上下文

**讲解重点：**

本课程中的 Agent 不是只回答问题的聊天工具，也不是独立承担经营责任的自治组织。它能够在获得允许的上下文后，按明确任务读取资料、修改文档或代码、运行验证并汇报结果。人负责目标、边界、审批和责任，这构成当前可以管理和落地的协作方式。

**转场：** 将它放进同一案例，可看到执行链上的新分工。

### `P19` 流程全景：人和 Agent 如何协作交付

**讲解重点：**

业务或产品负责人先确认目标和范围，技术责任人确保 Agent 获得足够且适当的上下文；Agent 依据确认规格执行任务并产出变更和验证结果；人审查修改是否合规、是否越界，再决定是否批准交付。变化集中在执行与反馈的密度，门禁仍然存在。

**转场：** 当执行更加迅速，输入给 Agent 的内容就会更直接影响结果。

### `P20` 为什么规格更重要：速度放大输入质量

**讲解重点：**

如果目标只存在于模糊聊天里，Agent 可能很快实现错误理解；如果排除范围没有明确写出，Agent 可能主动增加看似有用但未经批准的能力；如果没有验证标准，团队容易把“已经生成”误认成“已经交付”。因此规格、约束、任务与验证要求需要更显性。

**转场：** 这正是当前公开实践中常见的规格驱动方向。

### `P21` Spec-Driven Development：共同趋势与事实边界

**讲解重点：**

从三个公开项目可观察到的共同趋势，是将意图、规格、任务和验证环节明确组织起来，使 Agent 执行有可审查的依据。但它们的流程、侧重点和适用方式不同，不能据此得出一种统一标准，更不能笼统宣称传统 PRD、设计说明或审查材料必然消失。企业如何衔接既有文档，取决于风险和治理要求。

**转场：** 用客户反馈功能可以把这条规格驱动链路具体化。

### `P22` 场景中的规格驱动链路

**讲解重点：**

先写清业务目标是收集意见，再确认本次范围只有提交和查看；将通知、状态和统计写入排除约束；随后才把任务交给 Agent 执行，并要求以功能验证和范围检查作为交付证据。首尾两端均由人确认和批准，这使速度与控制可以同时成立。

**转场：** 接下来查看执行之前和执行之后分别应留下什么材料。

### `P23` 拟真产物一：上下文、确认规格与任务

**讲解重点：**

这一页模拟的是 Agent 接手任务前的工作包：业务目标告诉它为何做，确认范围告诉它做什么，禁止扩展告诉它不能做什么，任务与验证要求告诉它如何报告完成。与简单的一句提示相比，这些材料更容易评审、复用和纠偏。

**转场：** Agent 给出的结果也必须可审查，而不能只有一句完成声明。

### `P24` 拟真产物二：修改摘要、验证与批准

**讲解重点：**

修改摘要应直接对应确认范围，并明确说明没有加入被排除的功能。验证结果既检查业务能力，也检查是否越界。最终，负责人看到变更和证据后记录批准决定。这里强调的是交付证据的完整链条，不是认为 Agent 自己宣布成功就可以发布。

**转场：** 接下来以公开框架说明这种工程组织正在如何出现。

### `P25` 实践实例一：Spec Kit

**讲解重点：**

依据 Spec Kit 官方仓库，它以规格驱动开发为主张，公开工作流包括建立原则、形成规格、生成计划、拆解任务并实施。课程中引用它，是为了展示一种从意图到任务的结构化组织方式。这里不评价它是否适合某个具体团队，也不将其视为唯一标准。

**转场：** 另一种公开实践更强调对每项变化增加轻量规格层。

### `P26` 实践实例二：OpenSpec

**讲解重点：**

OpenSpec 官方资料强调轻量、迭代并支持既有系统中的变化组织。其变更文件夹包含提案、规格、设计和任务等产物，使团队在编码前对齐要做什么。它说明 Agent 协作不一定要求庞大的前置流程，也可以围绕单次变化保留必要依据。

**转场：** 第三个实例把注意力更多放在执行过程本身的纪律上。

### `P27` 实践实例三：Superpowers

**讲解重点：**

Superpowers 官方说明将其描述为面向编码 Agent 的软件开发方法体系，工作流包含前期澄清和设计确认、制定计划、执行、测试、评审以及完成分支等过程。它提供的启示是：即使执行由 Agent 加速，设计确认、测试和评审仍然可以作为强约束机制。

**转场：** 三种实例之后，回到管理人员最需要把握的责任控制链。

### `P28` 质量与责任：Agent 需要更清晰的控制链

**讲解重点：**

在 Agent 协作中，质量控制需要覆盖输入、执行和输出：人先批准规格和授权范围，Agent 在边界内执行，工具提供验证输出，人再审查变更与证据并作交付决定。Agent 可以承担大量操作，但权限越大、速度越快，审批人对范围和证据的核查越不能缺失。

**转场：** 下面不只讲正确路径，而是看一次真实可能发生的偏差。

### `P29` 失败与纠偏：发现越界后如何闭环

**讲解重点：**

案例已明确排除通知能力，但 Agent 可能因为认为“反馈后通知管理员更完整”而加入通知配置。这不是功能更丰富，而是越界。责任人发现后应指出偏差并重申约束，要求 Agent 删除越界内容，重新执行功能及范围验证，最后在证据充分时重新批准。课程通过这一闭环说明：Agent 模式要求更快发现并纠正错误，而不是容忍错误扩张。

**转场：** 三种模式已经分别成立，现在可以在同一坐标下比较。

## 6. 横向比较讲解

### `P30` 同一案例，三种组织路径

**讲解重点：**

三条路径均能交付相同的反馈功能，其差异主要是工作拆分和反馈发生的位置。阶段式以较完整的基线后续推进；迭代式让增量较早接受反馈；Agent 协作让明确任务在上下文中更快执行和验证。比较的目的不是排名，而是理解不同控制结构。

**转场：** 流程变化的背后，是人的工作重点与执行分配变化。

### `P31` 角色变化：谁做什么，谁负责什么

**讲解重点：**

在三种模式中，目标与发布责任都需要归属于人。变化在于执行组织：阶段式依赖专业环节交接，迭代式依赖跨职能协作和流水线，Agent 模式又增加了受控委派和结果审查。管理人员不能把 Agent 的执行贡献误读为责任主体转移。

**转场：** 角色变化也能从交付材料的形态中观察到。

### `P32` 产物变化：从阶段文件到执行证据

**讲解重点：**

阶段式以规范、设计和追踪矩阵支撑审查；迭代式以故事、验收、流水线和反馈待办支撑短周期交付；Agent 模式特别强调上下文约束、委派任务、修改摘要与工具验证。形式可以变化，但管理需要的事实始终包括：做了什么、为何这样做、如何证明正确。

**转场：** 这些产物最终都要回答相同的质量和责任问题。

### `P33` 质量与责任对照：门禁没有消失

**讲解重点：**

无论采用哪种方式，范围确认、方案审查、结果验证、发布批准和反馈回流都必须有人或机制负责。区别在于门禁所在的位置、验证发生的频率以及反馈循环的长度。Agent 可以让证据更快形成，也可能让错误更快传播，因此不能删除门禁。

**转场：** 因此选择协作方式应从适用条件出发，而不是赶潮流。

### `P34` 适用条件：不是三选一

**讲解重点：**

当范围稳定且追踪要求高，应强化基线和评审；当需要尽早验证价值，应强化迭代和反馈；当上下文能够明确提供、任务能够验证并且责任清晰时，可以引入 Agent 加快部分执行。现实组织往往组合这些能力，而不是只贴一种模式标签。

**转场：** 最后把 Agent 的机会与风险转化为管理者可以使用的检查问题。

## 7. 管理收束讲解

### `P35` Agent 的价值与风险同时存在

**讲解重点：**

Agent 的潜在价值来自缩短阅读、修改和验证之间的距离，但同一速度也会放大错误理解、范围越界和无证据完成。管理者真正要判断的不是“能不能使用 Agent”，而是团队是否具备把加速引导到正确结果上的控制能力。

**转场：** 第一组检查问题关注任务本身是否准备好。

### `P36` 管理者检查清单一：任务是否适合引入

**讲解重点：**

准备让 Agent 参与某项工作时，先判断目标能否说明、范围能否约束、上下文能否安全提供、结果能否分段验收、敏感权限能否隔离。若关键条件仍模糊，不意味着永远不能使用 Agent，而是应先把任务和治理输入整理清楚。

**转场：** 即使任务适合，组织还必须能审查结果并承担责任。

### `P37` 管理者检查清单二：组织是否能承担控制

**讲解重点：**

团队需要预先回答：谁批准规格，谁复核验证，谁拦截越界，谁批准发布，谁保留证据与决策记录。如果这些问题无人负责，Agent 的产出越多，组织风险反而越大。这份清单可用于内部讨论和后续治理设计的起点。

**转场：** 以一句主线判断结束三种模式的比较。

### `P38` 课程总结：模式演进的真正主线

**讲解重点：**

阶段式方法留下了基线、审查和追踪能力；敏捷与 DevOps 缩短了价值反馈距离并强化持续验证；Agent 协作使执行和验证循环进一步细化与加快。真正值得保留的结论是：工具能力可以提高效率，人的目标判断、边界批准和结果责任始终不可缺席。

**转场：** 最后一页提供事实来源，供课后进一步阅读。

### `P39` 参考资料与事实来源

**讲解重点：**

三项框架说明均依据其官方公开仓库资料整理。引用这些资料的目的，是证明当前已经存在围绕规格、变化管理和过程纪律组织 Agent 协作的公开实践，而不是进行选型推荐。培训材料后续如需要更新，应重新核查这些公开资料的最新表述。
