## 1. 引言

### 1.1 目的

本文档将已经确认的产品需求细化为可设计、可开发、可测试和可验收的软件需求。文档定义最小可行产品（MVP）的系统边界、用户角色、详细功能规则、输入校验、权限限制、工单状态约束、异常行为、数据留存和非功能要求。

本文档确认后，将作为页面与交互说明、技术设计说明书、数据模型设计、接口设计、测试与验收方案以及 MVP 实施计划的需求依据。

### 1.2 系统范围

系统为面向外部个人客户的售后工单系统，包含客户门户和内部后台。客户能够注册、登录、提交问题并通过公开留言跟踪处理过程；管理员能够管理分类和账号、分配并直接处理工单；客服能够查看工单并处理分配给本人的工单。

MVP 面向本地演示与小规模试用，使用浏览器访问，预计客户不超过 100 人、客服不超过 5 人。系统不在本阶段提供附件、站外通知、服务级别协议（SLA）、统计报表、高级检索、批量操作、客户确认关闭、重开工单或正式生产部署能力。

## 2. 总体描述

### 2.1 系统组成

系统应包含以下使用界面和服务能力：

| 组成部分 | 服务对象 | 主要能力 |
| --- | --- | --- |
| 客户门户 | 客户 | 注册、登录、创建工单、查看本人工单、公开留言 |
| 内部后台 | 客服、管理员 | 查看工单、处理工单、工单分配、分类与账号管理 |
| 后端服务 | 全部界面 | 身份认证、授权、业务校验、数据访问和记录保存 |
| 持久化组件 | 后端服务 | MVP 使用 JSON 文件保存数据，并支持未来更换存储实现 |

### 2.2 用户角色

| 角色 | 创建方式 | 数据访问范围 | 核心职责 |
| --- | --- | --- | --- |
| 客户 | 开放注册 | 仅本人账号和本人创建的工单 | 提交问题、查看状态、补充留言 |
| 客服 | 管理员创建 | 可查看全部工单，仅处理本人负责工单 | 回复客户、推进处理状态 |
| 管理员 | 系统初始化 | 可访问管理范围内全部数据与全部工单 | 配置分类、维护账号、分配和处理工单 |

### 2.3 运行约束

- 前端使用原生 `HTML`、`CSS` 与 `JavaScript`。
- 后端使用 Python `FastAPI` 提供 REST 接口。
- MVP 数据使用 `JSON` 文件持久化，但业务逻辑必须通过仓储抽象访问数据。
- 系统必须能够在本地环境启动并通过现代桌面浏览器使用。
- MVP 只要求支持小规模试用，不要求高并发或多实例部署。

### 2.4 范围外能力

以下能力不属于本 SRS 的 MVP 软件需求：

- 企业客户组织、租户隔离和多人共享客户工单。
- 邮箱验证、找回密码、自助修改密码、多因素认证和登录失败锁定。
- 工单附件、内部备注、邮件/短信/第三方消息通知。
- 客户确认关闭、状态退回、取消、重开、自动关闭。
- 统计看板、组合查询、关键词检索、导出和批量操作。
- 真实关系型数据库、正式部署、备份恢复和生产运维。

## 5. 认证与账号需求

### 5.1 客户注册

| 项目 | 说明 |
| --- | --- |
| 用例编号 | UC-AUTH-01 |
| 触发角色 | 未登录访客 |
| 前置条件 | 访客未登录；系统可正常保存数据。 |
| 输入信息 | 用户名、电子邮箱、密码、确认密码。 |
| 成功结果 | 系统创建角色为“客户”、状态为“启用”的账号；用户可使用其登录标识登录。 |
| 失败结果 | 系统不创建账号，并返回明确的校验或冲突提示。 |

#### 功能需求

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-AUTH-001 | 系统应向未登录访客提供客户注册能力。 | 必须 |
| SRS-AUTH-002 | 注册产生的账号角色必须固定为“客户”，访客不得在注册时选择客服或管理员角色。 | 必须 |
| SRS-AUTH-003 | 注册成功后，新客户账号默认为启用状态。 | 必须 |
| SRS-AUTH-004 | 用户名和电子邮箱在系统内分别必须唯一，比较时应忽略首尾空白；电子邮箱应采用不区分大小写的唯一性判断。 | 必须 |
| SRS-AUTH-005 | 密码和确认密码不一致时，系统不得创建账号。 | 必须 |

#### 注册字段约束

| 字段 | 是否必填 | 约束 | 错误情形 |
| --- | --- | --- | --- |
| 用户名 | 是 | 去除首尾空白后长度为 3 至 50 个字符；允许中文、英文字母、数字、下划线和连字符。 | 为空、长度不合法、字符不合法或已存在。 |
| 电子邮箱 | 是 | 去除首尾空白后符合基本邮箱格式；长度不超过 254 个字符。 | 格式不合法或已存在。 |
| 密码 | 是 | 长度为 8 至 128 个字符。MVP 不强制复杂度组合。 | 为空或长度不合法。 |
| 确认密码 | 是 | 必须与密码完全一致。 | 与密码不一致。 |

> 确认说明：已确认 PRD 仅规定开放注册和用户名/邮箱登录；SRS 已确认注册时用户名与电子邮箱均为必填项，并采用基础密码长度限制。

### 5.2 登录与退出

| 项目 | 说明 |
| --- | --- |
| 用例编号 | UC-AUTH-02 |
| 触发角色 | 客户、客服、管理员 |
| 前置条件 | 用户账号存在且为启用状态。 |
| 输入信息 | 登录标识、密码。 |
| 成功结果 | 系统建立已认证访问状态，并依据角色进入相应页面。 |
| 失败结果 | 系统不建立登录状态，并提示登录失败。 |

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-AUTH-006 | 系统应允许用户使用用户名或电子邮箱作为登录标识。 | 必须 |
| SRS-AUTH-007 | 登录时电子邮箱匹配应不区分大小写，用户名按保存值精确匹配。 | 必须 |
| SRS-AUTH-008 | 密码验证失败、账号不存在或账号被禁用时，系统不得建立认证状态。 | 必须 |
| SRS-AUTH-009 | 对外登录失败提示不得暴露账号是否存在或是否已禁用，应统一提示“账号或密码错误，或账号不可用”。 | 必须 |
| SRS-AUTH-010 | 已登录用户应能退出；退出后原认证状态不得继续访问受保护功能。 | 必须 |
| SRS-AUTH-011 | MVP 不提供找回密码、自助修改密码、邮箱验证或登录锁定能力。 | 必须 |

### 5.3 初始化管理员与客服账号管理

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-AUTH-012 | 系统应提供初始化首个管理员账号的机制；该机制不得允许普通访客在已有管理员后继续创建管理员。 | 必须 |
| SRS-AUTH-013 | 仅管理员能够创建客服账号。 | 必须 |
| SRS-AUTH-014 | 客服账号创建时应输入用户名、电子邮箱和初始密码，并遵守客户账号相同的唯一性及密码长度规则。 | 必须 |
| SRS-AUTH-015 | 管理员不得通过客户注册入口创建客服或管理员角色。 | 必须 |
| SRS-AUTH-016 | MVP 不要求管理员编辑、禁用客服账号或重置客服密码。 | 必须 |

### 5.4 客户账号启用与禁用

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-AUTH-017 | 管理员应能查看客户账号列表，列表至少展示用户名、电子邮箱、启用状态和创建时间。 | 必须 |
| SRS-AUTH-018 | 管理员应能将启用的客户账号设置为禁用，并将禁用账号重新设置为启用。 | 必须 |
| SRS-AUTH-019 | 被禁用客户不得登录、创建工单或追加留言。 | 必须 |
| SRS-AUTH-020 | 客户账号被禁用后，其历史工单、公开留言和操作关联信息不得被删除或隐藏于管理员查询范围之外。 | 必须 |

## 6. 问题分类需求

### 6.1 分类字段

| 字段 | 是否必填 | 约束 |
| --- | --- | --- |
| 分类标识 | 系统生成 | 全局唯一，创建后不得变化。 |
| 分类名称 | 是 | 去除首尾空白后长度为 1 至 50 个字符；在未删除的数据中不得重名。 |
| 状态 | 系统/管理员维护 | `启用` 或 `停用`；新建时默认为启用。 |
| 创建时间 | 系统生成 | 创建后不得修改。 |
| 更新时间 | 系统生成 | 编辑名称或状态时更新。 |

### 6.2 分类管理功能

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-CAT-001 | 仅管理员可以查看分类管理列表、创建分类、编辑分类名称或切换分类状态。 | 必须 |
| SRS-CAT-002 | 管理员创建分类时，系统应验证分类名称必填、长度和唯一性规则。 | 必须 |
| SRS-CAT-003 | 管理员可以停用分类；MVP 不提供物理删除分类功能。 | 必须 |
| SRS-CAT-004 | 客户创建工单时，系统只应返回并接受处于启用状态的分类。 | 必须 |
| SRS-CAT-005 | 分类停用前已创建的工单仍应展示原分类名称，不因分类停用而失去关联信息。 | 必须 |
| SRS-CAT-006 | 客服可在工单信息中查看分类，但不得维护分类。 | 必须 |

## 7. 工单需求

### 7.1 工单字段与约束

| 字段 | 产生方式 | 是否必填 | 约束 |
| --- | --- | --- | --- |
| 工单标识 | 系统生成 | 是 | 全局唯一，创建后不可更改。 |
| 标题 | 客户输入 | 是 | 去除首尾空白后长度为 1 至 100 个字符。 |
| 问题描述 | 客户输入 | 是 | 去除首尾空白后长度为 1 至 4000 个字符。 |
| 问题分类 | 客户选择 | 是 | 创建时必须引用一个有效分类。 |
| 创建客户 | 系统确定 | 是 | 固定为当前登录客户，不得由前端指定其他客户。 |
| 当前状态 | 系统维护 | 是 | 新建时为“待分配”；按状态规则更新。 |
| 当前负责人 | 管理员指定 | 否 | 未分配时为空；分配后引用有效客服账号。 |
| 创建时间 | 系统生成 | 是 | 创建后不可修改。 |
| 更新时间 | 系统生成 | 是 | 工单分配、留言或状态变化时更新。 |

> 确认说明：标题和问题描述的长度约束属于 SRS 补充的可验证输入规则，现已纳入设计和测试依据。

### 7.2 客户创建工单

| 项目 | 说明 |
| --- | --- |
| 用例编号 | UC-TKT-01 |
| 触发角色 | 已登录且启用的客户 |
| 前置条件 | 至少存在一个有效问题分类。 |
| 主流程 | 客户进入新建页；选择有效分类；填写标题和问题描述；提交；系统校验并创建工单。 |
| 成功结果 | 创建状态为“待分配”、负责人为空的工单；客户可在本人工单列表看到该工单；系统记录创建动作。 |
| 异常结果 | 信息不符合约束、分类不存在或已停用、账号不可用时，不创建工单并返回原因。 |

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-TKT-001 | 仅已登录且启用的客户能够创建工单。 | 必须 |
| SRS-TKT-002 | 工单创建输入只包含标题、问题描述和问题分类，不包含附件、紧急程度或期望时间等字段。 | 必须 |
| SRS-TKT-003 | 系统在保存工单前必须再次验证所选分类仍为有效状态。 | 必须 |
| SRS-TKT-004 | 创建成功时，系统应将工单状态设置为“待分配”，负责人设置为空。 | 必须 |
| SRS-TKT-005 | 工单创建后，MVP 不允许客户编辑标题、问题描述或分类。 | 必须 |

### 7.3 工单列表与详情查看

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-TKT-006 | 客户应能查看本人创建的工单列表，至少展示标题、分类、当前状态和创建时间。 | 必须 |
| SRS-TKT-007 | 客户应能打开本人创建的工单详情，并查看工单字段、状态和公开留言。 | 必须 |
| SRS-TKT-008 | 客户访问其他客户工单时，后端必须拒绝请求，不得返回该工单内容。 | 必须 |
| SRS-TKT-009 | 客服与管理员应能查看全部工单列表，至少展示标题、客户、分类、状态、负责人和创建时间。 | 必须 |
| SRS-TKT-010 | 客服与管理员应能按单一状态筛选内部工单列表；不要求其他筛选或关键词搜索。 | 必须 |
| SRS-TKT-011 | 客服与管理员应能查看任意工单详情、公开留言及基础操作记录。 | 必须 |

## 8. 工单分配需求

| 项目 | 说明 |
| --- | --- |
| 用例编号 | UC-ASG-01 |
| 触发角色 | 管理员 |
| 前置条件 | 工单存在；目标客服账号存在且可用于被分配。 |
| 主流程 | 管理员打开工单详情；选择客服；执行分配；系统保存负责人并记录操作。 |
| 成功结果 | 工单负责人更新；目标客服获得该工单处理权限；操作记录可查看。 |
| 异常结果 | 工单不存在或目标账号不是可分配客服时，分配失败且原负责人不变。 |

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-ASG-001 | 仅管理员可以为工单指定或更换负责人。 | 必须 |
| SRS-ASG-002 | 工单可以在任一未关闭状态下被首次分配或重新分配。 | 必须 |
| SRS-ASG-003 | 已关闭工单不得重新分配。 | 必须 |
| SRS-ASG-004 | 负责人必须是系统中存在的客服账号；不得将客户或管理员设为负责人。 | 必须 |
| SRS-ASG-005 | 分配或重新分配成功后，系统必须记录操作者、操作时间、变更前负责人和变更后负责人。 | 必须 |
| SRS-ASG-006 | 重新分配后，原客服立即失去该工单的留言和状态操作权限，但仍能查看工单。 | 必须 |
| SRS-ASG-007 | 工单分配本身不得自动改变工单状态。 | 必须 |

> 确认说明：基线已要求管理员可重新分配；SRS 进一步确认已关闭工单不可重新分配，以保护已完成的处理记录。

## 10. 工单状态需求

### 10.1 状态转换矩阵

| 当前状态 | 允许的下一状态 | 允许操作者 | 前置条件 |
| --- | --- | --- | --- |
| 待分配 | 处理中 | 管理员 | 无需已分配；管理员可直接介入处理。 |
| 待分配 | 处理中 | 负责客服 | 工单已分配给该客服。 |
| 处理中 | 已解决 | 管理员 | 无。 |
| 处理中 | 已解决 | 负责客服 | 工单仍分配给该客服。 |
| 已解决 | 已关闭 | 管理员 | 无。 |
| 已解决 | 已关闭 | 负责客服 | 工单仍分配给该客服。 |
| 已关闭 | 无 | 无 | 已关闭工单不可继续流转。 |

### 10.2 状态处理规则

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-STS-001 | 工单状态集合必须限定为“待分配”“处理中”“已解决”“已关闭”。 | 必须 |
| SRS-STS-002 | 新创建工单的初始状态必须为“待分配”。 | 必须 |
| SRS-STS-003 | 状态只能依据状态转换矩阵向下一阶段推进，不允许跳级、回退、取消或重开。 | 必须 |
| SRS-STS-004 | 未被分配为负责人的客服不得改变任意工单状态。 | 必须 |
| SRS-STS-005 | 管理员可以推进任意未关闭工单的状态，包括尚未分配负责人的待分配工单。 | 必须 |
| SRS-STS-006 | 状态改变必须记录操作者、操作时间、原状态和新状态。 | 必须 |
| SRS-STS-007 | 状态改变不得要求同时提交留言；留言和状态推进是独立操作。 | 必须 |
| SRS-STS-008 | 工单进入已关闭状态后，应禁止新增留言、重新分配与进一步状态操作。 | 必须 |

> 确认说明：SRS 将线性状态流程具体化为“不得跳级”，并确认留言、分配和状态变更均为显式独立动作。

## 14. 数据持久化与演进需求

| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| SRS-DAT-001 | MVP 应将用户、分类、工单、留言和操作记录持久保存于本地 `JSON` 数据文件中。 | 必须 |
| SRS-DAT-002 | 服务重新启动后，已经成功保存的数据应能够继续读取和使用。 | 必须 |
| SRS-DAT-003 | 数据访问应通过仓储接口或等效抽象实现，业务服务不得直接依赖 `JSON` 文件读写细节。 | 必须 |

| SRS-DAT-005 | 业务实体应使用稳定且唯一的标识符，以便后续迁移到关系型数据库时保留关联关系。 | 必须 |

| SRS-DAT-007 | JSON 至真实关系型数据库的数据迁移能力不属于 MVP 实现范围，但 MVP 数据结构和仓储边界不得阻碍后续迁移。 | 必须 |

## 19. 已确认的需求细化项

以下内容是在不扩大 MVP 功能模块的前提下，为使需求可实现和可测试而细化的规则；这些规则已经用户确认，并纳入正式需求基线：

| 编号 | 已确认细化项 | 规则 |
| --- | --- | --- |
| C-01 | 注册登录标识 | 注册时用户名和电子邮箱均必填且唯一；登录可使用任一标识。 |
| C-02 | 基础密码要求 | 密码长度为 8 至 128 个字符，MVP 不强制复杂度。 |
| C-03 | 文本输入长度 | 分类名称 1-50，标题 1-100，问题描述 1-4000，留言 1-2000 个字符。 |
| C-04 | 关闭后的不可变操作 | 工单关闭后禁止留言、重新分配和状态操作。 |
| C-05 | 显式处理动作 | 分配、留言与状态推进互不自动触发，由操作者显式执行。 |
| C-06 | 分配范围 | 管理员可在未关闭状态下分配/重新分配，仅能指派客服账号。 |

| C-08 | 被禁用客户会话 | 被禁用客户后续受保护请求应失效，历史信息继续保留。 |
| C-09 | 操作记录扩展 | 除工单流程记录外，同时记录分类变更与客户账号启停操作。 |
| C-10 | 工单提交后字段修改 | MVP 不支持客户修改已提交工单的标题、描述或分类。 |
