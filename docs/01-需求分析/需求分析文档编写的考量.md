# 需求分析文档编写的考量

## 问题

需求分析应该拆解为 PRD、SRS、页面与交互说明吗？有没有固定的写法？

## 结论

**写法不是固定的，适合的才是最好的。** 不应机械套用 PRD + SRS + 页面说明的三分法，而应根据团队规模、协作模式和变更频率来决定文档结构。

## 核心依据

**信息应该按照使用方式组织，而不是按照文档类型组织。**

### 依据一：认知负荷理论

人类工作记忆容量有限（Miller 定律：7±2 个组块）。开发者写某个功能时，需要同时记住该功能的所有规则、约束和 UI 行为。如果这些信息分散在多份文档里，来回跳转会显著增加认知负担。

### 依据二：高内聚原则

一个功能域的所有相关信息应该内聚在一起，而不是按文档类型拆分。按功能域组织时，"工单分配"的规则、谁能分配、状态怎么变、页面怎么显示、异常怎么处理，全部在一个连续段落里。

### 依据三：维护成本与团队规模正相关

当实际协作模式是"一个人做决策 + AI 辅助执行"时，PRD 给业务方、SRS 给开发、页面说明给前端这种拆分的理由不复存在。同一个决策者需要在三份文档之间追同一个规则，改一处就要改三处，维护成本超过阅读收益。

### 依据四：敏捷实践

Scrum 的 Product Backlog Item 和用户故事（INVEST 模型）都采用按功能域打包的方式——把一个功能的所有规格（前置条件、操作、结果、字段约束、权限规则、异常场景）放在一个地方。

## 推荐的文档结构（适用于小团队 / 一人 + AI 模式）

### 文档 1：产品定义（合并 PRD 的精华）

只保留不可推导的决策：为什么做、为谁做、不做什么、验收标准。不写功能列表的展开细节。

### 文档 2：功能规格（合并 SRS + 页面与交互）

按功能域/工作流组织，每个功能域包含：

```
## 工单处理
- 用户故事：作为客服，我需要...
- 业务规则：...
- 字段约束：...
- 权限：...
- 状态转换：...
- 页面行为：列表展示哪些字段、筛选条件、点击进入详情
- 异常场景：...
```

好处：一个功能的所有信息在一个连续段落里，开发者或 AI 写功能时不需要在多个文档间来回跳。

### 文档 3：技术设计说明书（独立保留）

架构、分层、持久化、安全、部署。这些是"怎么做"的决策，跟"做什么"的规格应该分开，因为技术变更不会影响产品意图，产品需求变更也不应该直接改技术方案。

## 反过来说：三分法什么时候合适

如果团队有明确的职责分工（产品经理、业务分析师、前端开发、后端开发、测试各有人负责），PRD + SRS + 页面说明的三分法是合适的——不同角色只看自己需要的部分，减少每个人的阅读负担。

**文档结构应该服务于实际的使用场景，而不是追求形式上的完整性。**

## 文档选择的决策矩阵

| 场景 | 推荐结构 |
|---|---|
| 一人 + AI / 2-3 人小团队 | 产品定义 + 功能规格 + 技术设计 |
| 10+ 人团队，角色分工明确 | PRD + SRS + 页面说明（三分法） |
| 需求频繁变更的项目 | 越精简越好，按功能域组织 |
| 交付给外部客户验收 | 保留完整的 PRD + SRS，形式更正式 |
