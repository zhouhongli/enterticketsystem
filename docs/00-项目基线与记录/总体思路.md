## 1. 关于项目目录结构是否符合主流设计方案

### 用户提问

> 项目目录结构是否符合现在的主流设计方案？

### 回答记录

结论：**总体符合现代 FastAPI 小型业务系统的常见设计思路，但建议在定稿前补充工程配置文件和 Python 包结构细节。** 当前方案不存在方向性问题，不需要推翻分层或改用复杂架构。

**符合之处**

- 《MVP 实施计划》采用 `FastAPI` 单体后端、原生前端、独立 `tests/` 和数据目录，适合当前 MVP 规模。
- `api / services / domain / repositories / security / storage` 的职责拆分与现有需求匹配，既避免所有逻辑堆在路由文件中，也没有引入微服务等过度设计。
- `repositories/` 与 `storage/` 分开是合理的：前者表达业务访问协议，后者处理 JSON 文件初始化、锁和原子替换，有利于未来更换数据库。
- `frontend/` 与 `backend/` 分开，同时由 FastAPI 同源提供静态资源，符合当前“无前端构建链”的方案。
- FastAPI 官方建议将较大应用拆分为 Python 包、`main.py`、依赖模块和多个 `APIRouter` 路由模块；本计划与该方向一致。

**建议调整**

1. 在 `backend/` 下明确增加 `pyproject.toml`。

   当前目录树只写了应用代码，没有列出 Python 项目配置文件。建议把后端视为独立 Python 应用工程，依赖、测试配置和 FastAPI 入口均由 `backend/pyproject.toml` 管理。

2. 明确增加 `__init__.py` 与具体路由文件。

   FastAPI 官方多文件结构以 Python package 组织模块。建议把 `api/routes/` 展开为 `auth.py`、`categories.py`、`admin.py`、`customer.py`、`internal.py`，并为 Python 包目录包含 `__init__.py`。

3. 增加开发配置与忽略文件。

   建议在根目录保留 `README.md`、`.gitignore`，在 `backend/` 增加 `.env.example`；其中应只提供管理员初始化等配置键示例，不保存真实密码或会话值。

4. 保持 `data/` 在静态资源目录之外。

   当前方案已经符合这一要求，应继续明确 `backend/data/store.json` 不得被 FastAPI 静态挂载访问。

**建议后的目录结构**

```text
enterpriseSpec/
  backend/
    pyproject.toml
    .env.example
    app/
      __init__.py
      main.py
      config.py
      api/
        __init__.py
        dependencies.py
        errors.py
        routes/
          __init__.py
          auth.py
          categories.py
          admin.py
          customer.py
          internal.py
      schemas/
      services/
      domain/
      repositories/
        protocols.py
        json_repository.py
      security/
      storage/
    data/
      .gitkeep
    tests/
      unit/
      service/
      api/
      storage/
  frontend/
    pages/
    assets/
      css/
      js/
  docs/
  README.md
  .gitignore
```

关于是否采用 Python 的 `src/` 布局：Python Packaging 官方说明其优点主要在于隔离可导入源代码并验证安装后的包；但本项目是本地运行的单体应用，而不是准备发布的通用 Python 包。因此，当前 `backend/app/` 布局是可接受且更直接的选择，无需为追求形式上的“主流”引入额外安装复杂度。

- [FastAPI 官方：Bigger Applications - Multiple Files](https://fastapi.tiangolo.com/tutorial/bigger-applications/)
- [Python Packaging 官方：src layout vs flat layout](https://packaging.python.org/en/latest/discussions/src-layout-vs-flat-layout/)
- [pytest 官方文档：Good Integration Practices](https://docs.pytest.org/en/stable/explanation/goodpractices.html)

## 2. 关于设计思路、瀑布模型、SDD 与 TDD

### 用户提问

> 请你再详细解释你的设计思路

### 回答记录

**整体思路**

我的设计思路不是先选择一个流程名称，再强行让项目符合它，而是从这个企业工单系统最容易出错、也最需要提前确定的事情出发，设计一套可实施、可追踪、可验收的开发路径。

这个系统虽然是 MVP，但它并不只是几个增删改查页面。它有三类用户、严格的访问边界、处理状态流、操作记录、会话认证、敏感信息隔离以及后续数据库替换需求。因此，真正重要的不是页面先做出来，而是先把下面这些规则钉牢：

- 客户只能访问自己的工单。
- 客服可以看全部工单，但只能处理分配给自己的工单。
- 管理员具有配置、分配和任意工单处理权限。
- 工单只能按固定状态流向前推进。
- 工单关闭后不能再被留言、分配或变更状态。
- 客户不能看到负责人或内部操作记录。
- 密码、会话凭证和内部信息不能泄露。
- JSON 只是 MVP 存储方式，业务逻辑不能被文件结构锁死。

所以，我采用的是“**先建立稳定业务基线，再按可验证增量实施**”的思路。

**为什么先做文档基线**

当前已经形成的文档不是为了形式完整，而是分别解决不同层面的问题：

| 层面 | 要解决的问题 | 当前文档 |
| --- | --- | --- |
| 产品层 | 做什么、不做什么、为谁解决问题 | PRD |
| 软件需求层 | 系统必须表现出什么行为 | SRS |
| 用户交互层 | 用户在哪些页面完成哪些操作 | 页面与交互说明 |
| 技术层 | 用什么结构实现这些行为 | 技术设计说明书 |
| 数据层 | 数据如何保存、约束和追溯 | 数据模型设计 |
| 接口层 | 前后端如何通信、如何鉴权和返回数据 | API 接口设计 |
| 验证层 | 如何证明设计没有漏掉需求 | 追踪矩阵/一致性审查记录 |
| 测试层 | 如何证明实现符合需求 | 测试与验收方案 |
| 执行层 | 以什么顺序开发并交付 | MVP 实施计划 |

这种划分的价值在于：每份文档回答一种问题，避免把产品目标、数据库字段、接口错误码和测试步骤混杂在一份巨大文档里。

例如，“客户不能看到负责人”这条规则会分别落在不同文档里：

| 文档 | 表达方式 |
| --- | --- |
| SRS | 客户只能访问允许的数据范围 |
| 页面与交互说明 | 客户详情页不显示负责人 |
| API 设计 | 客户详情响应不返回负责人字段 |
| 测试方案 | 检查客户响应和页面均无负责人信息 |
| 实施计划 | 前端展示和响应模型实现时纳入安全复核 |

这就是追踪性的意义：一个重要规则不是写过一次就算完成，而是必须贯穿需求、设计、实现和验证。

**为什么不是严格瀑布**

严格瀑布通常意味着：

```text
需求完全完成 -> 设计完全完成 -> 编码完成 -> 最后测试 -> 交付
```

这个顺序看起来规整，但对当前项目有两个问题。

第一，权限和状态等风险通常要在实现和测试中进一步暴露。如果直到最后才测试，发现“客户接口泄露内部字段”或“工单关闭后仍能写入”，返工成本会很高。

第二，MVP 的价值在于较快获得一个真实可运行的闭环。如果所有模块都写完后才首次集成，项目会长时间只有文档和孤立代码，没有可验证成果。

因此，我保留了瀑布模型中有价值的部分：

- 需求先确认。
- 设计先确认。
- 测试标准先确认。
- 开发前形成正式计划。
- 重要变更经过确认。

但在开发执行阶段，我不会采用一次性大交付，而是采用里程碑增量实现：

```text
M0 工程骨架
 -> M1 数据、存储、安全、会话基础
 -> M2 管理配置能力
 -> M3 客户建单能力
 -> M4 内部处理闭环
 -> M5 安全与页面完整联调
 -> M6 验收交付
```

每一个阶段都应该产生可运行代码和对应测试，而不是把测试推迟到最后。

**为什么更接近轻量 V 模型**

V 模型强调：左侧每一种定义和设计活动，都应在右侧有对应验证活动。这本质上就是 SDD 中规范与验证的对应关系。

当前项目可以这样对应：

```text
产品目标                 <-> 端到端业务验收
SRS 详细需求             <-> 功能与权限测试
页面与交互说明           <-> 浏览器交互验收
技术设计                 <-> 架构与安全审查
数据模型设计             <-> 持久化与完整性测试
API 接口设计             <-> API 集成测试
```

例如：

| 设计内容 | 对应验证 |
| --- | --- |
| 会话使用 `HttpOnly` Cookie | 检查 Cookie 属性、退出撤销和禁用后失效 |
| JSON 原子替换保存 | 模拟失败并检查无部分提交 |
| 负责人权限规则 | 非负责人客服执行写入必须被拒绝 |
| 状态线性流转 | 测试跳级、回退和关闭后写入均失败 |
| 分类名称快照 | 分类改名或停用后旧工单显示仍稳定 |

这也是为什么在编码开始前先完成《测试与验收方案》：测试不是设计完成后的附属品，而是设计是否成立的证明方式。

**SDD 在本项目中的位置**

`SDD` 在本项目中指 **规范驱动开发（Specification-Driven Development）**。

规范驱动开发的核心原则是：每一行代码都必须有明确的规范来源（需求文档、设计说明、接口定义等），并且每一项规范都必须在实现中有对应的代码。开发不是从"我觉得应该怎么做"出发，而是从"规范规定了什么行为"出发。

对于本项目，SDD 的规范体系由需求规范层和设计规范层共同构成：

| 规范层级 | 当前承载文档 | 回答的问题 |
| --- | --- | --- |
| 需求规范层 | PRD、SRS、页面与交互说明 | 系统应该**表现什么行为** |
| 设计规范层 | 技术设计说明书、数据模型设计、API 接口设计 | 系统**如何实现**这些行为 |

需求规范层定义系统的行为边界：PRD 确定业务范围和目标用户，SRS 逐条描述系统必须表现出的外部行为，页面与交互说明细化用户在哪些页面完成哪些操作。设计规范层定义系统的内部结构：技术架构、数据持久化方案、前后端通信协议。二者缺一不可——没有需求规范，设计就失去了依据；没有设计规范，实现就失去了指导。

页面与交互说明位于需求与设计之间，为前端实现提供直接输入。

采用规范驱动开发的原因有三点：

1. 重要规则（权限、状态流转、数据隔离）不依赖开发者的记忆或判断，而是有据可查。
2. 每次代码变更都可以追溯到对应的规范条目，避免"悄悄改规则"。
3. 测试方案不是随意编写的，而是直接来源于规范条目的验证用例。

不过，为了让项目治理更清晰，可以在实施计划中正式声明：

> PRD、SRS、页面与交互说明构成需求规范基线；
> 《技术设计说明书》《数据模型设计》《API 接口设计》构成设计规范基线；
> 二者共同组成本项目 MVP 的 SDD 规范基线。

这样后续开发人员知道：任何实现行为都应在这些文档中找到依据，而不仅仅是设计文档。

**SDD 中的验证层**

测试与验收不是规范本身，而是规范的验证层。在规范驱动开发中，规范的链条必须闭环：

```
需求规范（PRD / SRS / 页面与交互说明）
    ↓ 推导行为边界
设计规范（技术设计 / 数据模型 / API 接口设计）
    ↓ 指导内部实现
代码实现
    ↓ 必须符合规范
测试与验收 ← 证明实现符合规范
```

对应关系如下：

| 规范来源 | 对应验证 |
| --- | --- |
| PRD（产品需求） | 端到端业务验收：三类用户都能完成各自闭环 |
| SRS（软件需求） | 功能与权限测试：每条需求都有对应测试用例 |
| 页面与交互说明 | 浏览器交互验收：页面行为与文档描述一致 |
| 技术设计说明书 | 架构与安全审查：分层、隔离、密码、会话属性 |
| 数据模型设计 | 持久化与完整性测试：字段约束、快照、关联正确 |
| API 接口设计 | API 集成测试：路由响应、错误码、字段范围 |

验证层是 SDD 不可或缺的组成部分。没有它，规范就无法被证明是否被执行。规范定义了"应该怎么做"，验证证明"确实这么做了"。二者共同构成 SDD 的完整闭环。

**TDD 的两种含义**

这里要特别区分两个术语：

- `Technical Design Document`：技术设计文档。有些团队简称 TDD。
- `Test-Driven Development`：测试驱动开发。更常见的工程实践含义。

如果是前者，本项目已经覆盖：技术设计说明书、数据模型和 API 设计共同承担技术设计文档职责。

如果是后者，当前设计已经具备采用 TDD 的条件，但还没有把“所有代码都必须测试先行”写成硬性要求。

我不建议整个项目无差别地执行严格 TDD，因为不同模块的收益不同。

**适合严格 TDD 的部分**

这些模块一旦错误，会产生严重业务或安全问题，测试先行很有价值：

| 模块 | 为什么适合 TDD |
| --- | --- |
| 状态转换规则 | 规则离散清晰，容易先写允许/禁止路径测试 |
| 权限判断 | 越权是高风险问题，必须先固化拒绝行为 |
| 工单归属校验 | 客户越权读取是严重安全缺陷 |
| 分配与重新分配 | 涉及负责人变化和权限立即变化 |
| 关闭后限制 | 关系到数据不可变和处理闭环 |
| 会话失效 | 退出、过期、客户禁用后的行为必须可靠 |
| JSON 原子保存 | 必须验证失败不会留下部分数据 |
| 字段隔离 | 必须保证响应不会意外泄露内部数据 |

例如状态转换可先写测试：

```text
待分配 -> 处理中        允许
处理中 -> 已解决        允许
已解决 -> 已关闭        允许
待分配 -> 已解决        拒绝
已解决 -> 处理中        拒绝
已关闭 -> 任意状态      拒绝
```

然后再实现最小状态规则代码，使这些测试通过。

**不必强制严格 TDD 的部分**

以下内容更多属于展示和页面拼装，强制测试先行可能投入大于收益：

| 模块 | 更合适的方法 |
| --- | --- |
| 基础 HTML 页面布局 | 实现后浏览器检查 |
| CSS 样式 | 人工验收和视觉检查 |
| 静态导航外观 | 联调验证 |
| 普通文本提示细节 | 页面验收 |

但其中与安全或权限相关的页面行为仍需验证，例如无权限页面不能显示业务内容、用户输入不得作为脚本执行。

**实现时我会如何组织开发**

如果实施计划最终确认，我的开发顺序会是这样的：

1. `M0` 建立项目骨架

   创建 `backend/pyproject.toml`、FastAPI 应用、Python 包结构、测试目录、前端静态入口和配置示例。此时先证明项目可以启动、测试可以运行。

2. `M1` 建立最重要的底层约束

   先实现数据模型、仓储接口、JSON 存储、原子替换、密码哈希、服务端会话和管理员初始化。这一阶段重点采用测试先行，因为后面的所有业务都依赖这些基础。

3. `M2` 实现管理员基础配置

   管理员能够创建客服、管理分类、禁用或启用客户。没有这一步，客户建单和客服处理都缺少必要前提。

4. `M3` 实现客户门户闭环前半段

   客户注册、登录、选择有效分类、创建工单、查看本人详情、公开留言。这里会重点验证客户不能访问他人工单，也不能看到内部字段。

5. `M4` 实现内部处理闭环

   管理员分配工单，负责客服留言和推进状态，最终关闭工单；同时生成操作记录。这时系统具备完整业务价值。

6. `M5` 做安全复核和页面完整联调

   检查响应字段、脚本展示、Cookie、安全错误码、确认提示、排序、空状态和浏览器兼容。

7. `M6` 做正式验收

   执行完整自动化测试，并根据既定五条端到端场景演示系统，形成验收记录。

**变更控制思路**

文档先行不意味着文档不能变。真正重要的是：变化必须有依据、有影响评估、有确认记录。

实施中可能出现三类变化：

| 变化 | 处理方式 |
| --- | --- |
| 代码内部结构优化，不改变行为 | 直接实施并通过测试验证 |
| 接口字段、权限规则、页面行为变化 | 先更新对应基线并确认 |
| 想新增附件、报表、通知等能力 | 不塞入 MVP，记录为后续迭代候选 |

这样可以防止两种常见问题：

- 开发过程中悄悄扩大范围，导致 MVP 永远做不完。
- 为了赶进度，偷偷削弱权限、安全或验收标准。

**建议写入实施计划的方法声明**

当前实施计划已经描述了里程碑和测试同步落地，但还可以更明确地声明方法论：

```text
本项目采用规范驱动开发（SDD）与轻量 V 模型结合的方式组织 MVP 开发：
需求、设计和测试验收基线在编码前确认；
开发阶段按里程碑增量交付可运行能力；
每个里程碑同步完成相应验证。

PRD、SRS、页面与交互说明构成需求规范基线；
《技术设计说明书》《数据模型设计》《API 接口设计》构成设计规范基线；
《测试与验收方案》与自动化测试构成验证基线；
三者共同组成本项目 MVP 的 SDD 完整体系（规范 + 验证）。

对状态流转、权限与归属校验、会话失效、敏感字段隔离、
JSON 原子写入等高风险核心规则，采用测试驱动开发方式：
先形成失败测试，再实现最小可通过代码并进行重构。
页面布局和一般展示则采用实现后联调与浏览器验收方式。
```

这一补充不会改变系统功能，也不会让项目变重；它会让后续开发时“我们按什么规则工作”变得非常明确。

## 3. 关于任务拆解依据：业务流程还是代码层次结构

### 用户提问

> 任务拆解时，你是按照业务流程来的，还是按代码层次结构来的？

### 回答记录

我是采用**业务闭环主导、技术基础先行、代码分层落地**的混合拆解方式，不是单纯按业务流程，也不是单纯按代码层次结构。

**三个层次**

| 层次 | 作用 | 在实施计划中的体现 |
| --- | --- | --- |
| 技术基础层 | 先消除所有业务共同依赖的基础风险 | `M0` 工程骨架、`M1` 数据/仓储/安全/会话 |
| 业务能力层 | 按能够形成用户价值的流程增量交付 | `M2` 管理配置、`M3` 客户建单、`M4` 内部处理闭环 |
| 质量与交付层 | 对完整业务系统进行统一复核和验收 | `M5` 安全与联调、`M6` 验收交付 |

**为什么不完全按代码层次拆解**

如果严格按照代码层次开发，顺序可能是：

```text
全部数据模型
-> 全部仓储
-> 全部服务
-> 全部 API
-> 全部页面
-> 全部测试
```

这种方式在组织代码时很整齐，但有明显问题：很长时间内没有一条真正能使用的业务路径。直到所有层都基本完成，才能发现建单、分配、权限和页面行为是否真正连通。

此外，代码分层不代表交付价值。例如“完成所有路由文件”并不等于客户已经能够提交工单，也不等于客服能够完成处理闭环。

**为什么不完全按业务流程拆解**

如果完全沿着界面流程开发，顺序可能是：

```text
登录页面
-> 客户建单页面
-> 管理员分配页面
-> 客服处理页面
```

这也不稳妥，因为多个流程共用的底层规则如果尚未建立，就容易在各功能中临时拼凑：

- 密码哈希和会话认证需要先统一。
- 权限判断需要有统一方式。
- JSON 原子写入和操作记录必须从第一项业务写入开始就正确。
- 数据模型和仓储协议必须先确定，否则后续功能会重复返工。

因此，在能开始业务纵向交付之前，需要先完成一小段关键技术基础。

**当前拆解逻辑**

实施计划中的实际顺序是：

```text
M0 工程初始化与运行骨架
  建立可开发、可运行、可测试的工程环境

M1 数据、仓储、安全与会话基础
  建立后续业务都依赖的稳定底座

M2 管理基础配置能力
  先提供分类、客服账号和客户启停
  因为没有有效分类和客服，建单与分配流程无法完整运行

M3 客户工单能力
  实现注册、登录、选择分类、建单、查看与留言
  形成客户侧可用路径

M4 内部处理闭环
  实现查看、分配、处理、状态推进和关闭
  形成完整售后业务闭环

M5 安全、交互与完整联调
  统一检查跨角色、跨页面和跨接口问题

M6 验收交付
  按已经确认的测试方案证明 MVP 可交付
```

其中 `M2` 至 `M4` 基本按照业务依赖链展开：

```mermaid
flowchart LR
    A["管理员建立分类与客服"] --> B["客户提交工单"]
    B --> C["管理员分配工单"]
    C --> D["客服处理并留言"]
    D --> E["工单关闭并保留记录"]
```

而每个业务阶段内部，再按照代码层次落实。例如“客户创建工单”这项能力会同时涉及：

| 代码层次 | 对应实现 |
| --- | --- |
| `domain` | 建单字段约束、初始状态规则 |
| `repositories` | 保存工单与创建记录 |
| `services` | 校验客户身份、有效分类并组织一次提交 |
| `schemas` | 创建请求及客户详情响应字段 |
| `api/routes` | `POST /api/v1/customer/tickets` |
| `frontend` | 新建工单页与详情跳转 |
| `tests` | 正常建单、停用分类、越权字段和存储记录测试 |

也就是说，**不是先把某一层所有代码写完，再进入下一层，而是完成一条业务能力所需的各层最小纵向切片**。

**为什么这种方式适合本项目**

企业工单系统的核心价值不是某个单独模块，而是可控的处理闭环。按业务能力纵向实现有四个好处：

1. 每个阶段都有可以运行和验证的结果。
2. 权限、状态和审计规则会在实际流程中尽早接受验证。
3. 页面、接口、服务和存储不会长时间各自发展后才首次碰面。
4. 当 MVP 范围需要收紧时，可以围绕业务价值判断优先级，而不是围绕已经写了多少底层代码判断。

**与 TDD 的结合方式**

业务纵向拆解不妨碍测试先行。对于每个业务切片，我建议执行这样的内部顺序：

```text
识别该能力对应的需求与验收用例
-> 先写核心失败测试
-> 实现 domain / service 规则
-> 实现 repository 提交行为
-> 实现 API
-> 实现页面交互
-> 补充集成与浏览器验证
```

例如在 `M4` 的“重新分配工单”中，应先验证：

- 非管理员不能重新分配。
- 已关闭工单不能重新分配。
- 负责人只能是客服。
- 重新分配后原客服立即失去处理权限。
- 分配操作记录与负责人更新必须同时保存。

测试固定这些规则后，再实现对应的服务、仓储、接口和页面控件。

**简短概括**

我的拆解原则是：

> 先实现所有业务共用且风险最高的技术基础；再按完整业务流程拆分可交付增量；每项业务能力内部按代码层次纵向贯通，并用测试先行保护关键规则。

这比纯按层次开发更快获得可验证业务结果，也比纯按页面流程开发更能控制安全、数据一致性和后续演进风险。
