# 企业售后工单系统 产品与技术规范 v1.0

## 1. 产品定位

本系统是一套面向外部个人客户的售后工单系统。客户通过网页门户提交售后问题，内部客服与管理员在后台受理、回复、推进并关闭工单。首期 MVP 的目标是验证从问题提交到处理关闭的完整闭环，以及角色权限、数据持久化和后续技术演进路径，而不是直接建设功能完备的生产级客服平台。

MVP 面向小规模本地演示和试用场景：客户数量预计不超过 100 人，客服人员不超过 5 人。系统通过浏览器访问，使用本地可运行的服务完成演示。首期不包含复杂服务管理能力，但核心架构必须避免一次性实现，为后续统计分析、客服提效和正式部署保留扩展能力。

## 2. MVP 功能范围

客户可开放注册账号并使用用户名或邮箱、密码登录系统。登录后，客户能够查看自己创建的工单列表、创建新工单、进入工单详情页查看状态和双方公开留言，并继续追加留言。客户不得访问其他客户的账号信息或工单内容。

工单创建时仅要求填写标题、问题描述和问题分类。问题分类由管理员维护，客户只能选择当前有效的分类。MVP 不支持附件上传、外部消息通知、SLA、客户确认关闭、工单重开、高级搜索、导出或统计报表。

内部后台包含客服与管理员两类角色。管理员能够维护问题分类、创建客服账号、查看并启用或禁用客户账号、手动分配工单、重新分配工单，并可直接处理任意工单。客服可以查看工单列表并按状态筛选；对于已分配给自己的工单，客服可以公开回复客户并推进状态；对于未分配给自己的工单，客服仅能查看，不能留言或修改状态。

## 3. 工单流程与业务规则

每个新创建的工单初始状态为“待分配”。管理员可将其指定给某位客服，也可以直接介入处理。被指定为负责人的客服获得该工单的处理权限，其他客服仍保留只读查看权限。

MVP 状态流程固定为：

```text
待分配 -> 处理中 -> 已解决 -> 已关闭
```

负责客服或管理员可执行公开留言和状态推进操作。工单关闭由内部处理人员执行，不要求客户确认，也不提供自动关闭、取消、退回或重新打开能力。客户可以在工单未关闭期间继续追加公开留言，用于补充信息或反馈处理情况。

系统必须记录工单创建时间、当前状态、当前负责人、留言发送人及发送时间，并记录工单分配、重新分配和状态变化的操作人及时间。该记录用于还原基本处理过程，并作为后续统计与审计功能的数据基础。

## 4. 账号与权限规则

系统角色定义如下：

| 角色 | 主要能力 |
| --- | --- |
| 客户 | 注册、登录、创建工单、查看自己的工单、公开留言 |
| 客服 | 查看工单列表、按状态筛选、处理分配给自己的工单 |
| 管理员 | 管理客服与客户账号、管理分类、分配工单、处理任意工单 |

客户账号通过开放注册产生。MVP 支持基础登录和退出，不包含邮箱验证、找回密码、自助修改密码、多因素认证或登录失败锁定。被管理员禁用的客户账号不得继续登录或创建新工单，但历史工单与留言数据必须保留。

客服账号不得开放注册，由管理员在后台创建。系统启动时应提供初始化管理员的机制，以确保首次使用能够进入内部后台配置分类、账号和工单分配流程。

所有密码必须保存为安全哈希，不得以明文写入存储文件。后端接口必须基于当前登录用户身份实施权限验证，前端隐藏按钮不能替代后端授权检查。

## 5. 页面与交互范围

客户门户至少包含以下页面：

| 页面 | 功能 |
| --- | --- |
| 注册页 | 创建个人客户账号 |
| 登录页 | 客户、客服和管理员登录 |
| 我的工单页 | 查看当前客户创建的工单列表 |
| 新建工单页 | 填写标题、问题描述并选择分类 |
| 工单详情页 | 查看状态、公开留言并追加回复 |

内部后台至少包含以下页面：

| 页面 | 功能 |
| --- | --- |
| 工单列表页 | 查看全部工单并按状态筛选 |
| 工单详情页 | 查看工单信息、留言、分配与状态操作 |
| 分类管理页 | 新增、编辑、停用问题分类 |
| 客服账号管理页 | 创建和查看客服账号 |
| 客户账号管理页 | 查看、启用和禁用客户账号 |

前端采用原生 `HTML`、`CSS` 与 `JavaScript` 实现，不引入 React、Vue 等前端框架。页面以满足演示和业务验证为目标，不要求复杂组件体系或高级交互效果。

## 6. 技术架构

后端采用 `FastAPI` 提供 REST API，负责身份认证、权限判断、业务规则执行和数据访问协调。前端通过浏览器调用 API 完成注册、登录、工单处理和后台管理操作。

系统采用分层结构：

```text
原生 HTML/JavaScript 页面
        |
FastAPI REST API / 鉴权
        |
业务服务层
        |
仓储接口层
        |
JSON 文件存储适配器（MVP）
真实数据库适配器（后续）
```

业务服务层不得直接读取或修改 JSON 文件，而应通过统一仓储接口操作用户、分类、工单、留言与操作记录。MVP 使用 JSON 文件作为持久化介质，服务重启后账号和工单数据仍应保留。仓储接口应允许未来新增关系型数据库实现，而无需重写主要接口与工单业务规则。

JSON 适配器至少需要处理初始数据文件创建、唯一标识符生成、数据结构校验和写入保护。由于 MVP 面向小规模本地运行，不要求高并发能力，但不得因基本并发写入造成明显数据损坏风险。

## 7. MVP 验收标准

MVP 完成时，应能够演示以下完整业务路径：

1. 客户注册并登录系统。
2. 管理员创建并启用问题分类。
3. 客户选择分类，提交包含标题和问题描述的新工单。
4. 管理员查看待分配工单并指定负责客服。
5. 指定客服回复客户，将工单推进至处理中、已解决和已关闭。
6. 客户查看自己的工单状态和公开回复，并可在处理过程中追加留言。
7. 管理员可直接处理任意工单，也可禁用或启用客户账号。
8. 未被指定为负责人的客服不能修改该工单或发送处理回复。
9. 服务重启后，已有账号、分类、工单、留言和操作记录仍然存在。
10. 关键权限、密码哈希和状态流程具备自动化测试验证。

## 8. 已确定的后续迭代路线

MVP 完成后，按照以下顺序规划后续三轮迭代：

| 阶段 | 目标 | 核心范围 |
| --- | --- | --- |
| 迭代 1 | 管理看板 | 工单总量、状态分布、分类分布、客服处理量、处理时长统计 |
| 迭代 2 | 客服操作效率 | 多条件筛选、关键词搜索，并依据使用情况扩展快捷回复、批量操作或导出 |
| 迭代 3 | 正式运行基础 | 真实关系型数据库、数据迁移、部署配置、备份恢复和账号安全增强 |

第三轮完成后的功能方向暂不在当前规范中排定顺序，将根据真实使用反馈、工单数据与团队运营需求另行规划。
